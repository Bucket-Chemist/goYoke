## Executive Summary

The Claude Code CLI in `--output-format stream-json` mode has no mechanism to
request interactive tool approval from the TUI. This plan implements a
**PreToolUse hook** (`gogent-permission-gate`) that gates Bash commands through
the TUI's existing Permission modal via the UDS bridge. The CLI's
`--allowedTools` is expanded so the hook becomes the sole permission authority
for Bash, while all other tools are auto-allowed.

## Strategic Approach

- **Hook-based gate** — PreToolUse hooks fire before tool execution; the CLI
  blocks until the hook completes. No upstream modification needed.
- **Inverted permission model** — Instead of CLI-denies-then-hook-overrides
  (impossible), it's CLI-allows-then-hook-gates.
- **Bash-only scope** — Only Bash needs the permission modal. Write/Edit are
  handled by `--permission-mode acceptEdits`.
- **UDS bridge reuse** — The hook communicates with the TUI via the existing
  `GOFORTRESS_SOCKET` Unix domain socket.

## Success Criteria

1. Bash commands in TUI trigger a Permission modal (Allow / Deny / Allow for Session)
2. Allow → command executes normally
3. Deny → command blocked with descriptive error
4. Allow for Session → subsequent Bash commands auto-allowed
5. Read/Write/Edit/Glob/Grep → NO modal (auto-allowed)
6. Timeout (30s) → auto-deny with toast notification
7. Removing hook from settings.json + reverting allowedTools = full rollback
