# CM-003: Design Enrichment Schema, Mermaid Templates, and ARCHITECTURE.md Structure

**Phase**: Phase 0: Design & Evaluation
**Priority**: HIGH
**Blocked By**: CM-002 (extraction schema must be finalized first — this ticket extends it)
**Effort**: 1 day

## Description

Design the schemas and templates for Stage 2 (LLM enrichment) and Stage 3 (synthesis) of the pipeline. This ticket covers three deliverables:

1. **Enrichment schema**: Extends extraction schema with LLM-generated semantic fields
2. **Mermaid templates**: Standard diagram templates for each diagram type
3. **ARCHITECTURE.md structure**: Template for the human-readable output document

## Implementation Instructions

### Deliverable 1: Enrichment Schema

Create `.claude/schemas/codebase-map/enrichment.schema.json`.

The enrichment schema extends the extraction schema by adding these fields to each `Symbol`:

```json
{
  "enrichment_fields": {
    "description": {
      "type": "string",
      "description": "Human-readable 1-2 sentence description of what this symbol does",
      "examples": ["Validates Task() invocations against routing schema, blocking opus-tier calls and wrong subagent_types"]
    },
    "module_identity": {
      "type": "string",
      "enum": ["core", "helper", "glue", "config", "test", "generated"],
      "description": "Role of this symbol within its module",
      "definitions": {
        "core": "Primary business logic, the reason this module exists",
        "helper": "Utility function used by core, not meaningful alone",
        "glue": "Connects modules together (adapters, wrappers, converters)",
        "config": "Configuration loading, constants, defaults",
        "test": "Test code",
        "generated": "Auto-generated code"
      }
    },
    "complexity": {
      "type": "string",
      "enum": ["low", "medium", "high"],
      "description": "Cognitive complexity assessment",
      "definitions": {
        "low": "<10 lines, linear control flow, obvious purpose",
        "medium": "10-50 lines, branching/loops, requires reading to understand",
        "high": ">50 lines, complex state, concurrency, non-obvious algorithms"
      }
    },
    "is_entrypoint": {
      "type": "boolean",
      "description": "Whether this symbol is a primary entry point (main, handler, exported API)"
    },
    "architectural_role": {
      "type": "string",
      "enum": ["handler", "service", "repository", "middleware", "utility", "model", "factory", "orchestrator", "validator", "none"],
      "description": "Architectural pattern this symbol implements. Use 'none' if no pattern applies."
    },
    "tags": {
      "type": "array",
      "items": { "type": "string" },
      "description": "Cross-cutting concern tags",
      "examples": [["concurrency", "mutex"], ["io", "filesystem"], ["serialization", "json"]]
    }
  }
}
```

And these fields to the module-level object:

```json
{
  "module_enrichment_fields": {
    "module_description": {
      "type": "string",
      "description": "1-3 sentence description of the module's purpose and responsibilities"
    },
    "module_category": {
      "type": "string",
      "enum": ["command", "library", "internal", "test", "config", "documentation"],
      "description": "What kind of module this is"
    },
    "key_types": {
      "type": "array",
      "items": { "type": "string" },
      "description": "The 1-5 most important type/class names in this module"
    },
    "key_functions": {
      "type": "array",
      "items": { "type": "string" },
      "description": "The 1-5 most important function names in this module"
    }
  }
}
```

### Deliverable 2: Mermaid Templates

Create `.claude/schemas/codebase-map/mermaid-templates/`:

**Template 1: Module Dependency Graph** (`module-dependencies.mmd.tmpl`)

```mermaid
graph TD
    %% Auto-generated by /codebase-map
    %% Module dependency graph for {{.RepoName}}
    %% Generated: {{.Timestamp}}

    {{range .Modules}}
    {{.ID}}["{{.Name}}<br/><small>{{.Category}}</small>"]
    {{end}}

    {{range .Dependencies}}
    {{.From}} --> {{.To}}
    {{end}}

    %% Styling by category
    classDef command fill:#e1f5fe
    classDef library fill:#f3e5f5
    classDef internal fill:#e8f5e9
    {{range .Modules}}
    class {{.ID}} {{.Category}}
    {{end}}
```

**Template 2: Type Relationships** (`type-relationships.mmd.tmpl`)

```mermaid
classDiagram
    %% Type relationships for module: {{.ModuleName}}

    {{range .Types}}
    class {{.Name}} {
        {{range .Fields}}
        {{.Visibility}}{{.Name}} {{.Type}}
        {{end}}
        {{range .Methods}}
        {{.Visibility}}{{.Name}}({{.Params}}) {{.Returns}}
        {{end}}
    }
    {{end}}

    {{range .Relationships}}
    {{.From}} {{.Arrow}} {{.To}} : {{.Label}}
    {{end}}
```

Arrow types:
- `-->` = dependency (uses)
- `--|>` = implements interface
- `--*` = composition (has field of type)
- `--o` = aggregation (has slice/map of type)

**Template 3: Call Graph** (`call-graph.mmd.tmpl`)

```mermaid
graph LR
    %% Call graph for module: {{.ModuleName}}
    %% Showing functions with 2+ connections

    {{range .Functions}}
    {{.ID}}["{{.Name}}"]
    {{end}}

    {{range .Calls}}
    {{.Caller}} --> {{.Callee}}
    {{end}}

    %% Highlight entry points
    classDef entrypoint fill:#fff3e0,stroke:#e65100
    {{range .Entrypoints}}
    class {{.ID}} entrypoint
    {{end}}
```

## Mermaid Diagram Scoping Rules

For large repos, diagrams must be scoped to stay readable:

| Diagram Type | Scoping Rule |
|-------------|-------------|
| Module dependencies | Always full-repo (modules, not files) |
| Type relationships | Per-module. If >20 types, show only exported + key_types. |
| Call graph | Per-module. If >30 functions, show only entrypoints + 2-hop neighbors. |
