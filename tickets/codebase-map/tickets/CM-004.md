# CM-004: Design Storage Layout and Incremental Strategy

**Phase**: Phase 0: Design & Evaluation
**Priority**: MEDIUM
**Blocked By**: none
**Effort**: 0.5 days

## Description

Define the directory layout for codebase map artifacts, the `manifest.json` schema for tracking extraction state, and the incremental update strategy that allows re-running `/codebase-map` on only changed files.

## Implementation Instructions

### Deliverable 1: Storage Layout

```
.claude/codebase-map/                        ← Agent-accessible (not committed)
├── manifest.json                            ← Extraction metadata and timestamps
├── extract/                                 ← Raw extraction output (Stage 1)
│   ├── cmd--gogent-validate.json            ← Module path with -- separator
│   ├── cmd--gogent-archive.json
│   ├── internal--routing.json
│   └── ...
├── enriched/                                ← Enriched output (Stage 2) — clean separation
│   ├── cmd--gogent-validate.json
│   ├── cmd--gogent-archive.json
│   ├── internal--routing.json
│   └── ...

docs/codebase-architecture/{repo_name}/      ← Human-readable (committed)
├── ARCHITECTURE.md                          ← Main architecture document
├── diagrams/                                ← Standalone Mermaid files
│   ├── module-dependencies.mmd
│   ├── cmd--gogent-validate-types.mmd
│   ├── cmd--gogent-validate-callgraph.mmd
│   └── ...
└── manifest.json                            ← Copy of extraction manifest (for reference)
```

**Naming convention**: Module paths use `--` as separator (e.g., `cmd/gogent-validate` → `cmd--gogent-validate.json`) to avoid nested directories.

**Repo name resolution** (in priority order):
1. Git remote: `basename $(git remote get-url origin .git)`
2. Directory name: `basename $(git rev-parse --show-toplevel)`
3. Fallback: `"unknown-repo"`

### Deliverable 3: Incremental Update Strategy

**Full map flow** (`/codebase-map` or `/codebase-map --full`):

```
1. gogent-codebase-extract --path=. --output=.claude/codebase-map/extract/
   → Extracts ALL source files, overwrites all extract/*.json
2. Update manifest.json with new timestamps for all modules
3. LLM enriches ALL modules → overwrites .claude/codebase-map/*.json
4. LLM synthesizes → overwrites docs/codebase-architecture/
5. Update manifest.json: last_full_map = now
```

**Incremental flow** (`/codebase-map --incremental`):

```
1. Read manifest.json → get per-file extracted_at timestamps
2. git diff --name-only <last_commit> HEAD → get changed files
   OR: compare file mtime > extracted_at for each tracked file
3. Group changed files by module
4. gogent-codebase-extract --files=<changed_files> --output=.claude/codebase-map/extract/
   → Re-extracts only changed modules
5. MERGE: For each changed module:
   a. Read existing extract/{module}.json
   b. Replace file entries for changed files
   c. Add entries for new files
   d. Remove entries for deleted files
   e. Write updated extract/{module}.json
6. LLM re-enriches only changed modules
7. LLM re-synthesizes (reads ALL enriched modules, regenerates ARCHITECTURE.md)
8. Update manifest.json: last_incremental = now, per-file timestamps
```

**Staleness detection**:
- A module is "stale" if ANY of its files have `mtime > extracted_at`
- The Go binary flags stale modules in its output
- The fork skill can choose to re-enrich stale modules only

**New file detection**:
- Files not in manifest.json → new files, must be extracted
- Directories not mapping to any module → new modules

**Deleted file handling**:
- Files in manifest but not on disk → mark as deleted
- If all files in a module are deleted → remove module from manifest
- Remove corresponding enriched JSON and diagrams
