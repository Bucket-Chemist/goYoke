# CM-001: Evaluate Tree-sitter Go Bindings and Select Library

**Phase**: Phase 0: Design & Evaluation
**Priority**: HIGH
**Blocked By**: none
**Effort**: 1 day

## Description

Two Go binding libraries exist for tree-sitter. This ticket evaluates both, builds a proof-of-concept extraction, and selects the library for all subsequent work.

**Candidates:**

| Library | Maintainer | Notes |
|---------|-----------|-------|
| `github.com/smacker/go-tree-sitter` | Community | Well-established, bundled grammars, widely used |
| `github.com/tree-sitter/go-tree-sitter` | Official tree-sitter org | Official bindings, may have better long-term support |

### Step 2: Evaluation Criteria

Test each library against:

| Criterion | Weight | How to Measure |
|-----------|--------|---------------|
| **Grammar availability** | HIGH | Does it bundle Go, Python, TypeScript, R grammars? |
| **API ergonomics** | MEDIUM | Lines of code to extract function signatures |
| **Query support** | HIGH | Does it support tree-sitter query syntax (S-expressions)? |
| **Build complexity** | MEDIUM | CGo requirements, cross-compilation issues |
| **CGo-free build possible?** | HIGH | Can the library work without CGo (e.g., WASM bindings)? GOgent-Fortress has zero CGo deps today. |
| **Performance** | LOW | Time to parse 1000 Go files |
| **Maintenance** | MEDIUM | Last commit date, open issue count, release cadence |

### Step 3: Proof of Concept

Write a minimal extractor that:
1. Parses `fixtures/sample.go` with tree-sitter
2. Uses a query to find all function declarations
3. Extracts: function name, parameter list, return types, line number range
4. Outputs JSON matching the schema from CM-002

**Tree-sitter query for Go functions:**

```scheme
;; Function declarations
(function_declaration
  name: (identifier) @func.name
  parameters: (parameter_list) @func.params
  result: (_)? @func.returns) @func.def

;; Method declarations
(method_declaration
  receiver: (parameter_list) @method.receiver
  name: (field_identifier) @method.name
  parameters: (parameter_list) @method.params
  result: (_)? @method.returns) @method.def

;; Type declarations (struct, interface)
(type_declaration
  (type_spec
    name: (type_identifier) @type.name
    type: (_) @type.body)) @type.def

;; Interface declarations
(type_declaration
  (type_spec
    name: (type_identifier) @iface.name
    type: (interface_type) @iface.body)) @iface.def
```

### Step 4: Evaluate Pure-Go Alternatives

Before committing to CGo, assess whether a CGo-free path exists:

| Alternative | Feasibility | Notes |
|-------------|------------|-------|
| WASM-based tree-sitter | Check if Go WASM runtime can load tree-sitter grammars | Avoids CGo entirely, but may have performance overhead |
| `go/ast` for Go + tree-sitter for others | Hybrid approach | Go extraction uses native `go/ast` (zero CGo), only other languages need tree-sitter |
| Pure-Go parsers per language | Evaluate `go/ast`, Python/TS AST in Go | Maximum portability, but more code to maintain |

**If CGo is unavoidable**, document:
- Build system changes required (CGO_ENABLED=1, C compiler dependency)
- Cross-compilation impact
- Impact on existing `make install-*` targets
- CI pipeline changes needed
