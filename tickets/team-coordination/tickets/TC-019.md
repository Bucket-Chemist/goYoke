## Description

TC-015's entire Phase 2 design depends on the Claude Agent SDK supporting concurrent `query()` calls from the same Node.js process. This is explicitly "UNANSWERED" in TC-015. If the SDK doesn't support concurrent queries, the TUI concurrency fix needs a completely different approach (separate Node.js child processes per query, or an SDK feature request to Anthropic).

This ticket is a focused investigation to answer that question before any TC-015 implementation work begins.

### Why This Matters

The current TUI implementation has a critical concurrency bug: spawning Mozart while another query is active results in corrupted stdin/stdout because `query()` calls from `useClaudeQuery.ts` may share session state. TC-015 proposes a session management system to handle multiple concurrent queries, but that design assumes the SDK supports concurrent queries within a single Node.js process.

**If the SDK doesn't support concurrency, TC-015 needs a fundamentally different architecture.**

### Test 1: Concurrent query() Calls

Test if the SDK supports multiple concurrent `query()` calls from the same process.

```typescript
import { query } from '@anthropic-ai/claude-agent-sdk';

async function testConcurrentQueries() {
  const stream1 = query({ prompt: "Count to 10 slowly" });
  const stream2 = query({ prompt: "Count to 5 slowly" });

  try {
    const [result1, result2] = await Promise.all([
      collectStream(stream1),
      collectStream(stream2),
    ]);

    console.log("✓ Concurrent queries completed successfully");
    console.log("Result 1:", result1);
    console.log("Result 2:", result2);
    return "CONCURRENT_SUPPORTED";
  } catch (error) {
    console.error("✗ Concurrent queries failed:", error);
    return error.message.includes("session") ? "SINGLE_SESSION" : "ERROR";
  }
}

async function collectStream(stream: AsyncIterable): Promise<string> {
  const chunks: string[] = [];
  for await (const chunk of stream) {
    if (chunk.type === 'text') {
      chunks.push(chunk.content);
    }
  }
  return chunks.join('');
}
```

**Expected Outcomes:**

| Behavior | Meaning |
|----------|---------|
| Both complete without error | Concurrent queries supported ✓ |
| One errors with "session busy" | Single-session semantics |
| One queues behind the other | Sequential execution only |
| One stream gets corrupted | Shared state conflict |

### Test 2: Does a Second query() Cancel the First?

Test if starting a new query cancels an in-progress query.

```typescript
async function testQueryCancellation() {
  const results = { stream1: null, stream2: null };

  const stream1 = query({ prompt: "Count to 100 slowly" });

  // Start collecting stream1
  const promise1 = (async () => {
    try {
      results.stream1 = await collectStream(stream1);
      console.log("Stream 1 completed:", results.stream1);
    } catch (error) {
      console.error("Stream 1 cancelled:", error);
      results.stream1 = "CANCELLED";
    }
  })();

  // After 1 second, start stream2
  await new Promise(resolve => setTimeout(resolve, 1000));

  const stream2 = query({ prompt: "Say hello" });
  const promise2 = (async () => {
    try {
      results.stream2 = await collectStream(stream2);
      console.log("Stream 2 completed:", results.stream2);
    } catch (error) {
      console.error("Stream 2 failed:", error);
      results.stream2 = "ERROR";
    }
  })();

  await Promise.all([promise1, promise2]);

  return results;
}
```

**Expected Outcomes:**

| Behavior | Meaning |
|----------|---------|
| Stream 1 completes, Stream 2 completes | Concurrent queries work |
| Stream 1 cancelled, Stream 2 works | New query cancels old query |
| Stream 1 completes, Stream 2 queued | Sequential execution |
| Both error | SDK doesn't support this usage |

## Implementation Notes

### SDK Version to Test

Check `/home/doktersmol/Documents/GOgent-Fortress/packages/tui/package.json` for exact `@anthropic-ai/claude-agent-sdk` version.

### Testing Environment

- Run tests in isolated Node.js process
- Use actual SDK, not mocks
- Test with both simple and complex prompts
- Test with both short and long-running queries

### What to Look For

**Positive Signs** (concurrent queries work):
- Both streams complete without errors
- No "session busy" or "already active" errors
- Output from each stream is clean and isolated
- Event handlers for each stream fire independently

**Negative Signs** (concurrent queries don't work):
- "Session already active" errors
- Second query cancels first query
- Interleaved/corrupted output from streams
- Shared state causes one query to affect another
- Event stream ref gets overwritten

### If Concurrent Queries Don't Work

Document in TC-015 the alternative approach:

**Alternative: Process Isolation with IPC**

```
Router Process (main TUI)
│
├─► Query Process 1 (child_process.fork)
│   └─ Claude SDK instance 1
│
├─► Query Process 2 (child_process.fork)
│   └─ Claude SDK instance 2
│
└─► IPC Channel (structured messages)
    ├─ Query requests →
    ├─ Stream events ←
    └─ Cancellation signals ↔
```

Each child process runs its own SDK instance with isolated stdin/stdout. The main TUI process orchestrates via IPC, eliminating shared state issues.
