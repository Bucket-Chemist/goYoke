## Description

Migrate the `/review` slash command from foreground MCP-spawn orchestration to background `gogent-team-run` execution. This is the simplest of the three orchestrator rewrites because:

- No user interview phase (unlike braintrust)
- Fixed member count (1-4 reviewers, unlike implementation's variable members)
- Single wave (no inter-wave scripts needed)
- No helper binary needed (LLM generates config directly)

The review-orchestrator LLM agent is eliminated in the background path. The router generates config.json + stdin files directly from git diff output, then launches `gogent-team-run`.

**Reference:** `tickets/team-coordination/ReviewOrch-team-bridge.md` (authoritative corrections for TC-013's review section)

### Binary Infrastructure Fixes (Applied 2026-02-08)

The following bugs were discovered during live validation testing and fixed:

- [x] **`agents-index.json` flag splitting**: All 32 agents had `"--permission-mode delegate"` as a single string. `exec.Command` passes each argv element separately, so the CLI received one arg `--permission-mode delegate` instead of two args `--permission-mode` `delegate`. Agents crashed immediately (exit code 1 in <0.5s). **Fixed**: split to `"--permission-mode", "delegate"`.
- [x] **`spawn.go` missing `--model` flag**: `buildCLIArgs()` did not pass `--model` from the agent config. Agents defaulted to the session model (Opus) instead of their configured model (haiku for reviewers). **Fixed**: added `--model` to arg construction.
- [x] **`spawn.go` + `cost.go` CLI output parsing**: `claude --output-format json` outputs a JSON **array** of event objects, not a single object. `extractCostFromCLIOutput` tried `json.Unmarshal` into `map[string]interface{}` which failed. **Fixed**: new `parseCLIOutput()` function handles the array format, extracts cost from the `type: "result"` entry.
- [x] **`spawn.go` stdout file writing**: After an agent completed, nobody wrote the stdout file. The agent's response stayed in the captured CLI output buffer. **Fixed**: new `writeStdoutFile()` function in `finalizeSpawn()` extracts the agent's result text from the CLI JSON array and writes it to the stdout file path. Three-tier extraction: (1) try parsing result as JSON object, (2) try extracting ```json code block, (3) fallback: wrap text in `{"agent": ..., "raw_output": true, "result": "..."}`.

### BLOCKER: Agent Stdout Format Compliance

**Status: RESOLVED (2026-02-08)**

**Problem (was):** Reviewer agents responded in natural markdown text because `buildPromptEnvelope()` had no output format instructions.

**Fix applied:** `buildPromptEnvelope()` in `envelope.go` now:
1. Accepts `workflowType string` parameter (from `TeamConfig.WorkflowType` via `prepareSpawn()`)
2. Calls `resolveStdoutSchema(workflowType, agentID)` — candidate-based file lookup:
   - Exact: `{workflow}-{agentID}.json`
   - Suffix-stripped: strips `-reviewer`, `-critical-review`, `-pro`, `-writer`, `-archivist`
   - Worker fallback: `{workflow}-worker.json`
3. Embeds the `stdout` portion of the matched contract schema in the prompt
4. Adds "Expected Output Format" section: "Your response MUST be a single JSON code block" + schema + `$schema` field instruction
5. `writeStdoutFile()` tier-2 (```json code block extraction) parses the response — no changes needed downstream

**Test coverage:**
- `TestBuildSchemaCandidates` — 4 subtests verify candidate generation for reviewer/exact/critical-review/pro suffixes
- `TestBuildEnvelope_StdoutSchemaEmbedding` — verifies schema embedding with fake schemas dir
- `TestBuildEnvelope_NoSchemaAvailable` — verifies graceful degradation (no schema = no output format section)
- `TestResolveStdoutSchema_RealSchemas` — tests against real `~/.claude/schemas/` directory (review-standards, review-backend, braintrust-einstein, nonexistent)
- `TestResolveStdoutSchema_FakeDir` — tests resolution with overridden schemas dir

**Remaining:** Needs live end-to-end validation — run a review team and verify `stdout_*.json` contains structured JSON with `findings[]`, `$schema`, etc.

## Implementation Instructions

### 1. Router-Level Config Generation

The `/review` skill (or its router integration) must:

1. Capture git diff (`git diff --staged` or user-specified scope)
2. Classify changed files by language/domain
3. Select reviewers based on file types:
   - `.go`, `.py` backend files -> `backend-reviewer`
   - `.tsx`, `.ts` UI files -> `frontend-reviewer`
   - All files -> `standards-reviewer` (always)
   - 5+ files or cross-module -> `architect-reviewer`
4. Create team directory
5. Generate config.json from `schemas/teams/review.json` template
6. Generate stdin files per reviewer (must comply with `schemas/stdin/reviewer.json`)
7. Launch `gogent-team-run "$team_dir"`
8. Verify launch (read `background_pid` from config.json)
9. Return to user with team info

### 2. Config.json Generation

Copy `.claude/schemas/teams/review.json` template, populate:
- `team_name`: `"review-{timestamp}"` (timestamp format: `YYYYMMDD-HHmmss`)
- `project_root`: absolute path from `git rev-parse --show-toplevel`
- `session_id`: from `$GOGENT_SESSION_DIR` env var (set by TUI), NOT `$GOGENT_SESSION_ID`
- `created_at`: UTC ISO-8601 timestamp
- `waves[0].members[]`: only include selected reviewers (not always all 4)
- `budget_max_usd`: 2.0 (default from template)

Per-member fields that must be populated:
- `model`: `"haiku"` for all reviewers (code-reviewer tier)
- `timeout_ms`: `120000` (2 minutes per reviewer)
- `max_retries`: `1`
- `stdin_file`: `"stdin_{agent-name}.json"` (e.g., `stdin_backend-reviewer.json`)
- `stdout_file`: `"stdout_{agent-name}.json"`

**Team directory:** Create under `$GOGENT_SESSION_DIR/teams/{timestamp}.review/`

### 3. Stdin File Generation

Each reviewer gets a stdin file compliant with `schemas/stdin/reviewer.json`. Required fields:
- `agent`: enum from `["backend-reviewer", "frontend-reviewer", "standards-reviewer", "architect-reviewer"]`
- `workflow`: `"review"` (const)
- `context`: `{project_root, team_dir}` (absolute paths)
- `review_scope`: `{files[], total_files, languages_detected}` — NOTE: schema requires `files[]`, not `changed_files`. The architect contract now includes both.
- `git_context`: `{commit_message}` (+ optional `branch_name`, `related_tickets`)
- `focus_areas`: object (domain-specific per reviewer type, generic `type: object` in schema — LLM flexibility is acceptable here)
- `project_conventions`: object (language, conventions_file)
- `description`: string — **required by envelope builder** (not by schema, but by `buildPromptEnvelope` at runtime). Use format: `"Review {domain} code changes for {focus_areas_summary}"`

See `ReviewOrch-team-bridge.md` Section 3 for complete validated example. Note: bridge doc examples should conform to the formal schema; if discrepancies exist, the schema (`schemas/stdin/reviewer.json`) is authoritative.

### 4. Launch Command

```bash
gogent-team-run "$team_dir"
```

NO output redirection (binary handles `runner.log`). NO `config.json` path (binary takes directory).

### 5. Feature Flag

Support `settings.json -> "use_team_pattern": true/false`. When false, fall back to current foreground MCP-spawn pattern (review-orchestrator agent).

### 6. Result Retrieval

`/team-result` (TC-012) aggregates reviewer stdout files:
1. Read `stdout_*.json` from team directory
2. Extract findings from each reviewer
3. Group by severity — **use 5-level uppercase**: CRITICAL, HIGH, MEDIUM, LOW, INFO (canonical scheme per `/team-result` SKILL.md)
4. Compute approval status (APPROVED/WARNING/BLOCKED)
5. Display formatted report

**Severity alignment:** Reviewer stdout contracts currently use 3-level lowercase (`critical|warning|info`). Update contract stdout schemas to use 5-level uppercase to match `/team-result` parsing expectations. This prevents silent mismatches during aggregation.

## Notes

This is the first team-run workflow to be implemented. It serves as validation for the entire pattern. Issues discovered here inform TC-013b (braintrust) and TC-013c (implementation).

Risk mitigation: Start with manual testing (real `/review` usage during development). Automated e2e tests follow after manual verification.

### Decision Point (Post-Implementation)

**LLM config generation reliability:** Measure the schema validation pass rate of LLM-generated config.json + stdin files during implementation. Record the rate.

- If **>=70%**: Proceed to TC-013b with LLM-based config generation (same pattern).
- If **<70%**: Design a Go helper binary for config generation before starting TC-013b. The braintrust workflow's multi-wave complexity makes unreliable config generation a higher risk.

This measurement is referenced by TC-013b as a pre-condition.

- **D-1 schema change safe:** Adding `description` to stdin schemas did not break existing `go test ./cmd/gogent-team-run/...` (verified Phase 0).
