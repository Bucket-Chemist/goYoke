## Description

TC-013 covers 3 orchestrator rewrites (Mozart, ReviewOrchestrator, ImplManager) and already contains substantial inline implementation detail. However, TC-013 was written before several key tickets completed, creating mismatches between its inline examples and actual implementations:

- **TC-008** delivered `gogent-team-run` with a different CLI interface than TC-013 assumes
- **TC-009** delivered schemas with different naming conventions than TC-013 references
- **TC-010** delivered `gogent-team-prepare-synthesis` (not referenced in TC-013)
- **TC-018** delivered the full Mozart interview protocol (already integrated into `mozart.md`)
- **TC-019** confirmed SDK supports concurrent queries (affects execution path choices)

This ticket produces **bridge documents** that reconcile TC-013's design with actual implementations, plus a decision matrix for when to use each execution path.

### What Changed from Original TC-020

The original TC-020 proposed 3 separate design docs with full prompt templates. This was **redundant** because:

1. **Mozart**: TC-018 already delivered the interview protocol, config generation logic, and stdin templates. These are already integrated into `mozart.md` (Phases 2-2.5). A separate Mozart-rewrite.md would duplicate existing work.

2. **TC-013 evolution**: TC-013 now contains detailed step-by-step implementation instructions for all 3 workflows, including config population tables, stdin generation, launch commands, and 17 test cases. Separate design docs would largely repeat TC-013 content.

3. **Interface mismatches**: The original TC-020 assumed CLI flags (`--config`, `--session-dir`, etc.) that don't exist. The actual binary takes a single positional argument.

The reworked ticket focuses on what's actually missing: **bridging the gap** between TC-013's stale inline examples and the implementations that now exist.

### Target: Background Team Execution

| Orchestrator | New Dispatch Mechanism | New Agent Spawning |
|-------------|----------------------|-------------------|
| **Mozart** | Router calls `Task(opus)` for interview → generates config → launches `gogent-team-run <team-dir>` | Go binary spawns `claude` CLI processes per wave member |
| **Review-Orchestrator** | Router generates config directly → launches `gogent-team-run <team-dir>` | Go binary spawns `claude` CLI processes per reviewer |

### Two Execution Paths Coexist

TC-019 confirmed the SDK supports concurrent `query()` calls. This means two valid execution paths exist:

| Path | Mechanism | When to Use |
|------|-----------|-------------|
| **Foreground (MCP)** | `mcp__gofortress__spawn_agent` from within a Task()-spawned agent | Interactive workflows, small teams, when user needs real-time feedback |
| **Background (Go binary)** | `gogent-team-run <team-dir>` launched via Bash | Large teams, long-running workflows, when TUI should return immediately |

The decision matrix (Deliverable 3) documents when each path is appropriate.

## Actual Binary Interface (Source of Truth)

### `gogent-team-run`

```bash
# Usage (from cmd/gogent-team-run/main.go)
gogent-team-run <team-directory>

# The binary:
# 1. Reads config.json from <team-directory>/config.json
# 2. Creates PID file at <team-directory>/gogent-team-run.pid
# 3. Redirects output to <team-directory>/runner.log
# 4. Executes waves sequentially, members within waves in parallel
# 5. Updates config.json with status, PIDs, costs, timestamps
```

**There are NO named flags.** No `--config`, `--session-dir`, `--team-name`, `--output-file`.

### `gogent-team-prepare-synthesis`

```bash
# Usage (from cmd/gogent-team-prepare-synthesis/main.go)
gogent-team-prepare-synthesis <team-directory>

# The binary:
# 1. Reads stdout_einstein.json and stdout_staff-arch.json from team dir
# 2. Extracts key sections with graceful degradation
# 3. Writes pre-synthesis.md to team dir for Beethoven
```

Referenced in config.json as `waves[0].on_complete_script`.

### Config.json Structure (from `config.go`)

```go
type TeamConfig struct {
    TeamName            string  `json:"team_name"`
    WorkflowType        string  `json:"workflow_type"`
    ProjectRoot         string  `json:"project_root"`
    SessionID           string  `json:"session_id"`
    CreatedAt           string  `json:"created_at"`
    BudgetMaxUSD        float64 `json:"budget_max_usd"`
    BudgetRemainingUSD  float64 `json:"budget_remaining_usd"`
    WarningThresholdUSD float64 `json:"warning_threshold_usd"`
    Status              string  `json:"status"`
    BackgroundPID       *int    `json:"background_pid"`
    StartedAt           *string `json:"started_at"`
    CompletedAt         *string `json:"completed_at"`
    Waves               []Wave  `json:"waves"`
}

type Wave struct {
    WaveNumber       int      `json:"wave_number"`
    Description      string   `json:"description"`
    Members          []Member `json:"members"`
    OnCompleteScript *string  `json:"on_complete_script"`
}

type Member struct {
    Name         string  `json:"name"`
    Agent        string  `json:"agent"`          // NOT "agent_id"
    Model        string  `json:"model"`
    StdinFile    string  `json:"stdin_file"`     // Relative to team dir
    StdoutFile   string  `json:"stdout_file"`    // Relative to team dir
    Status       string  `json:"status"`         // pending|running|completed|failed
    ProcessPID   *int    `json:"process_pid"`
    ExitCode     *int    `json:"exit_code"`
    CostUSD      float64 `json:"cost_usd"`
    CostStatus   string  `json:"cost_status"`    // ok|warning|exceeded
    ErrorMessage string  `json:"error_message"`
    RetryCount   int     `json:"retry_count"`
    MaxRetries   int     `json:"max_retries"`
    TimeoutMs    int     `json:"timeout_ms"`
    StartedAt    *string `json:"started_at"`
    CompletedAt  *string `json:"completed_at"`
}
```

**Key field corrections vs TC-013 inline examples:**
- `agent` not `agent_id`
- `budget_max_usd` not `budget_total_usd`
- `process_pid` not `pid`
- Members are nested in `waves[].members[]`, not flat with a `wave` field

### 3. `tickets/team-coordination/team-dispatch-decision-matrix.md`

**Purpose**: Decision matrix documenting when to use each execution path. Answers the architectural question: "Given a workflow, should it use MCP spawn, Task(), or gogent-team-run?"

**Must include:**

1. **Execution path comparison**

   | Criterion | MCP spawn_agent | Task() + Foreground | gogent-team-run (Background) |
   |-----------|----------------|--------------------|-----------------------------|
   | **TUI blocks?** | Yes (parent waits) | Yes (parent waits) | No (returns in <15s) |
   | **Cost visibility** | Real-time in parent | Real-time in parent | Post-hoc from config.json |
   | **Error handling** | Parent catches errors | Parent catches errors | Errors in runner.log + config.json |
   | **User interaction** | Possible (AskUserQuestion) | Possible (AskUserQuestion) | None after launch |
   | **Max concurrency** | Limited by MCP | Limited by Task() nesting | Limited by budget + CPU |
   | **Available at Level 1+** | Yes | No (blocked by hooks) | Yes (via Bash) |
   | **Process isolation** | Separate CLI process | Separate CLI process | Separate CLI process + session leader |
   | **Progress tracking** | Manual | Manual | `/team-status` (TC-012) |
   | **Cancellation** | Kill PID | Context cancellation | `/team-cancel` (TC-012) |

2. **Per-workflow recommendation**

   | Workflow | Recommended Path | Rationale |
   |----------|-----------------|-----------|
   | **Braintrust** (full) | **Background** (gogent-team-run) | 3 Opus agents = 5-7 min; TUI must not freeze |
   | **Braintrust** (Einstein only) | **Foreground** (MCP spawn) | Single agent, user may want to interact |
   | **Code Review** (4 reviewers) | **Background** (gogent-team-run) | 4 parallel agents, 2-3 min; user can work meanwhile |
   | **Code Review** (quick, 1 reviewer) | **Foreground** (Task/MCP) | Single agent, <30s, overhead of team-run not worth it |
   | **Implementation** (multi-wave) | **Background** (gogent-team-run) | Variable duration, parallel waves |
   | **Implementation** (single task) | **Foreground** (Task()) | Direct delegation, no team overhead |

3. **Decision flowchart**

   ```
   How many agents will run?
     │
     ├─ 1 agent → Foreground (Task or MCP spawn)
     │
     ├─ 2-3 agents, total <2 min → Either (user preference)
     │
     └─ 3+ agents OR >2 min → Background (gogent-team-run)
         │
         Does it need user interaction during execution?
         │
         ├─ Yes → Foreground interview first, then background dispatch
         │         (Mozart pattern: Task(opus) for interview → team-run for execution)
         │
         └─ No → Direct background dispatch
                 (Review pattern: router generates config → team-run)
   ```

4. **Fallback strategy**
   - If `gogent-team-run` binary is missing: fall back to MCP spawn pattern
   - Feature flag: `settings.json → "use_team_pattern": true/false`
   - Both paths produce same output format (same files, same locations)

5. **Cost comparison**
   - Background path adds ~$0 marginal cost (Go binary, no LLM tokens)
   - Foreground MCP path adds ~$0.01-0.05 per spawn (envelope overhead)
   - For 4+ agents, background is strictly cheaper (no orchestrator LLM maintaining state)
