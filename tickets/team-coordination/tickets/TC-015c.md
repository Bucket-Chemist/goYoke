## Description

Full refactor of `useClaudeQuery.ts` to support multiple concurrent `query()` calls with independent streaming state. This is the original TC-015 Phase 2/3 design, deferred after Braintrust analysis determined team-run (TC-013a/b/c) solved the acute orchestration freezes.

**Current status:** TC-019 confirmed the Claude Agent SDK supports concurrent `query()` calls. The foundation is ready. This ticket is deferred because TC-015a (input buffer) captures 90% of the user value at 5% of the cost.

**When to undefer:** Promote this ticket when any of these conditions emerge:
- Users frequently need parallel foreground queries (not just queued messages)
- An interactive workflow requires two simultaneous conversations
- The TUI needs to handle multiple Mozart interviews concurrently
- Team-run proves insufficient for some orchestration pattern

---

## Design (preserved from original TC-015)

### Core Changes

1. **Per-query state model:** Replace global `isStreaming` / `streamingRef` / `currentMessageRef` with `Map<string, ActiveQuery>`
2. **Remove streaming guard:** Line 600 `if (streamingRef.current) return;` replaced with concurrent query creation
3. **Query-scoped event handlers:** All event handlers receive `queryId` parameter
4. **Zustand store:** `UISlice.streaming: boolean` → `UISlice.activeQueryCount: number`
5. **Per-query eventStreamRef:** Each query owns its eventStream for `streamInput()` (built-in tools)

### Known Hazards

| Hazard | Impact | Mitigation |
|--------|--------|------------|
| `eventStreamRef` sharing | `streamInput()` targets wrong query's stream | Per-query eventStream lookup |
| `currentMessageIdRef` interleaving | Concurrent queries mix message IDs | Per-query message buffer |
| Modal queue disambiguation | AskUserQuestion from 2 queries simultaneously | Add query context to modal header |
| React Map immutability | Mutation instead of new Map in `setActiveQueries` | Enforce `new Map(prev)` pattern |
| Cleanup race condition | Query completes before setState batches | `finally` block guarantees cleanup |

### Implementation Details

See original TC-015.md, Phase 2 (sections 2.1-2.4) and Phase 3 for complete refactoring steps, code examples, and store changes.

## Success Criteria (from original TC-015)

- [ ] User can type new messages while a query is streaming
- [ ] Two concurrent `sendMessage()` calls both complete successfully
- [ ] Each query's messages appear in correct order in chat history
- [ ] Tool results attributed to correct assistant message
- [ ] Built-in tools show modal for each query independently
- [ ] No race conditions in stress testing
- [ ] Memory usage stable with 5+ concurrent queries
- [ ] Existing single-query workflow still works correctly
- [ ] Status line shows accurate active query count
- [ ] Interrupt (Ctrl+C) stops ALL active queries gracefully
