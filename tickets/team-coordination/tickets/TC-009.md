# TC-009: Design All Three Team Templates

**Phase**: Phase 1: Schema Design

**Priority**: HIGH
**Blocked By**: none
**Effort**: 2-3 days

## Pre-Implementation Notes (2026-02-07)

**TC-006 pulled forward the following TC-009b scope:**
- Created `implementation.json` team config template with full field parity
- Created `implementation-worker.json` stdin/stdout schema
- Fixed nested budget → flat budget in review.json and implementation.json specs (below)
- Renamed `agent_id` → `agent` and `pid` → `process_pid` across all templates to match TC-008 Go struct
- Added missing TC-008 fields: `warning_threshold_usd`, `status`, `started_at`, `completed_at`, `exit_code`, `error_message`, `cost_status`, wave `description`
- Added `context.project_root`/`team_dir` to all stdin-stdout schemas

**Remaining TC-009 scope:** Formal JSON Schema `.schema.json` validation files, schema versioning, extended validation rules.

## Description

Design and write the complete schema set for all three workflow templates:

1. **Braintrust** (`braintrust.json`): Mozart interview-then-background. 3 members (einstein, staff-arch, beethoven), 2 waves, inter-wave synthesis script. Already partially specified in the implementation plan.

2. **Review** (`review.json`): Fully backgroundable. 4 reviewers (backend, frontend, standards, architecture) in 1 wave. Router fills stdin from git diff. No inter-wave script.

3. **Implementation** (`implementation.json`): Fully backgroundable. Dynamic members from specs.md task DAG. Multiple waves based on `blocked_by` relationships. Router fills stdin from ticket descriptions.

For each template, deliver:
- Team config template (`.claude/schemas/teams/{workflow}.json`)
- Stdin schema per agent type (`.claude/schemas/stdin/{agent}.json`)
- Stdout schema per agent type (`.claude/schemas/stdout/{agent}.json`)

### Team Config Schema Structure

All three team config templates follow this common structure:

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://gogent-fortress/schemas/teams/{workflow}.json",
  "title": "{Workflow} Team Configuration",
  "version": "1.0.0",
  "description": "{Workflow-specific description}",

  "team_name": "{workflow}-{timestamp}",
  "workflow_type": "{braintrust|review|implementation}",
  "project_root": "/absolute/path/to/project",
  "session_id": "{session-uuid}",
  "created_at": "{ISO-8601-timestamp}",

  "budget_max_usd": 5.0,
  "budget_remaining_usd": 5.0,
  "warning_threshold_usd": 4.0,

> **Standardization Decision (Review Item 9)**: Budget fields are **flat top-level** (`budget_max_usd`, `budget_remaining_usd`), NOT nested under a `"budget"` object. This resolves the cross-ticket contradiction between TC-008 (flat), TC-009 (was nested), and TC-013 (different names).

  "waves": [
    {
      "wave_number": 1,
      "description": "{wave purpose}",
      "members": [
        {
          "name": "{agent-id}",
          "agent_id": "{agent-id}",
          "model": "{haiku|sonnet|opus}",
          "stdin_file": "stdin_{agent-id}.json",
          "stdout_file": "stdout_{agent-id}.json",
          "status": "pending",
          "pid": null,
          "exit_code": null,
          "cost_usd": 0.0,
          "retry_count": 0,
          "max_retries": 1,
          "timeout_ms": 300000
        }
      ],
      "on_complete_script": null
    }
  ],

  "background_pid": null,
  "status": "pending",
  "heartbeat_file": "heartbeat.txt",
  "heartbeat_interval_seconds": 30,
  "started_at": null,
  "completed_at": null
}
```

### File Naming Conventions (Review Item 10)

All stdin/stdout files use `member.name` as the key, which is always unique within a team:

| Workflow | Naming Scheme | Example | Rationale |
|----------|--------------|---------|-----------|
| Braintrust | `stdin_{member.name}.json` | `stdin_einstein.json` | member.name == agent role (unique per team) |
| Review | `stdin_{member.name}.json` | `stdin_backend-reviewer.json` | member.name == reviewer domain |
| Implementation | `stdin_{member.name}.json` | `stdin_task-001.json` | member.name == ticket ID (unique) |

**Rules:**
- Stdin: `stdin_{member.name}.json`
- Stdout: `stdout_{member.name}.json`
- `member.name` is unique within a team
- For braintrust/review: `member.name` matches `agent_id`
- For implementation: `member.name` is the task ID

> **Note**: TC-008's Go binary uses `member.Name` (from config.json `name` field) to construct file paths. Ensure consistency between template names here and the Go struct.

## Code Guidelines

### Schema Naming Conventions

1. **File names**: Lowercase with hyphens: `backend-reviewer.json`, `staff-architect.json`
2. **Field names**: Snake_case for all JSON fields: `project_root`, `wave_number`, `stdin_file`
3. **Agent IDs**: Match agents-index.json exactly: `einstein`, `staff-architect-critical-review`, `backend-reviewer`
4. **Enum values**: Lowercase with underscores: `braintrust`, `completed`, `high_priority`

### Required Fields

All stdin schemas MUST include:
- `agent`: Agent ID from agents-index.json
- `role`: Brief description of agent's role in this workflow
- `workflow`: Workflow type (braintrust | review | implementation)
- `context.project_root`: Absolute path to project (for tool resolution)
- `reads_from`: Object with absolute paths to input files
- `expected_output.schema`: Reference to stdout schema

All stdout schemas MUST include:
- `agent`: Agent ID
- `status`: Completion status (completed | failed | partial)
- `wave`: Wave number
- `timestamp`: ISO-8601 timestamp
- `content`: Structured deliverable
- `metadata`: Token usage, quality metrics

### Absolute vs Relative Paths

- **ALL file paths in schemas**: Absolute paths (e.g., `/home/user/project/file.go`)
- **Go binary resolves**: Router/planner fills in absolute paths at template instantiation time
- **Agent reads**: Agents receive absolute paths, use Read tool directly

### Schema Validation

Use JSON Schema validators:
```bash
# Install ajv-cli
npm install -g ajv-cli

# Validate a filled template
ajv validate -s .claude/schemas/teams/braintrust.json -d /path/to/filled-config.json

# Validate stdin file
ajv validate -s .claude/schemas/stdin/einstein.json -d /path/to/stdin_einstein.json
```

### Consistency Patterns

1. **Budget structure**: Same across all team configs (budget_max_usd, budget_remaining_usd, warning_threshold_usd)
2. **Member structure**: Same across all workflows (name, agent_id, model, stdin_file, stdout_file, status, pid, cost_usd, etc.)
3. **Envelope structure**: All stdout files have agent/status/wave/timestamp/content/metadata
4. **Severity levels**: CRITICAL > HIGH > MEDIUM > LOW (consistent casing)

### Optional Fields in Stdin Schemas (Review Item 30)

The following fields are OPTIONAL and may not be present depending on workflow configuration:

| Field | When Present | When Absent |
|-------|-------------|-------------|
| `reads_from.scout_metrics` | Scout was run during planning | No scout was requested |
| `reads_from.wave1_outputs` | Wave 2+ agents only | Wave 1 agents |
| `reads_from.pre_synthesis` | Beethoven only | All other agents |

**Agent instruction** (include in prompt envelope boilerplate):
```
If optional fields in your stdin are missing, proceed with whatever context is available.
Do NOT error or ask for missing optional fields.
```

**JSON Schema pattern** for optional sections:
```json
{
  "required": ["agent", "task", "context", "paths"],
  "properties": {
    "reads_from": {
      "description": "Optional file references. Not all fields will be present.",
      "properties": {
        "scout_metrics": { "type": "string", "description": "OPTIONAL: path to scout_metrics.json" },
        "wave1_outputs": { "type": "array", "description": "OPTIONAL: Wave 1 stdout file paths" }
      }
    }
  }
}
```

## Dependencies and Integration Points

### Feeds Into TC-008 (Go Binary)

The Go binary's `readTeamConfig()` function will:
1. Parse `config.json` from one of these three templates
2. Extract `project_root` and set `cmd.Dir` accordingly
3. Read stdin files from `.claude/schemas/stdin/{agent}.json` filled paths
4. Write to stdout files and validate against `.claude/schemas/stdout/{agent}.json` envelope

The `buildPromptEnvelope()` function will:
1. Read the stdin file (e.g., `stdin_einstein.json`)
2. Construct the prompt with:
   - Agent orientation (role, workflow, wave)
   - Context (project_root, team_dir)
   - Task specification
   - Absolute paths to input files
   - Expected output format reference

### Feeds Into TC-013 (Orchestrator Rewrites)

Mozart, review-orchestrator, and impl-manager will:
1. Copy the appropriate template (braintrust.json, review.json, implementation.json)
2. Fill in:
   - `project_root`: Resolve from session context
   - `session_id`: Current session UUID
   - `team_name`: Generate timestamp
   - `stdin_file` paths: Write absolute paths to stdin JSON files
3. Write filled config.json to team directory
4. Launch `gogent-team-run <team-dir>`

### Inter-Wave Script (TC-010)

`gogent-team-prepare-synthesis` will:
1. Read `stdout_einstein.json` using the stdout schema
2. Read `stdout_staff-architect.json` using the stdout schema
3. Extract structured sections (root cause, challenged assumptions, findings)
4. Write `pre-synthesis.md` for Beethoven
5. This markdown file is referenced in `stdin_beethoven.json` as `reads_from.pre_synthesis`
