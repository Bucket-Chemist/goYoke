# TC-013: Rewrite Orchestrator Prompts for Team Pattern

**Phase**: Phase 4: Orchestrator Rewrites

**Priority**: HIGH

**Blocked By**: TC-012 (Slash Commands)

**Effort**: 3-5 days

## Description

This is the Phase 4 capstone ticket that rewrites all three orchestrator workflows (Mozart/braintrust, review-orchestrator/review, impl-manager/ticket) to use the team pattern with the `gogent-team-run` Go binary. This integrates all components built in Phases 1-3 and delivers the core value proposition: TUI returns to user within 30 seconds while orchestration continues in the background.

Each workflow has a different interaction pattern:
- **Mozart (interview-then-background)**: User interview phase is foreground, then dispatch team and return
- **Review (immediate background)**: Router builds config, launches team, returns in <10s
- **Impl-Manager (immediate background)**: Router parses specs.md, launches waves, returns in <10s

All three workflows must preserve the existing output quality while dramatically reducing TUI freeze time.

##### Project Root Detection (Review Item 17)

Mozart (and all orchestrators) must resolve `project_root` at planning time using this priority:

```go
func resolveProjectRoot() (string, error) {
    // Priority 1: Explicit env var (set by TUI or user)
    if dir := os.Getenv("GOGENT_PROJECT_DIR"); dir != "" {
        if _, err := os.Stat(dir); err == nil {
            return dir, nil
        }
        return "", fmt.Errorf("GOGENT_PROJECT_DIR=%s does not exist", dir)
    }

    // Priority 2: Git root (works from any subdirectory)
    out, err := exec.Command("git", "rev-parse", "--show-toplevel").Output()
    if err == nil {
        root := strings.TrimSpace(string(out))
        if root != "" {
            return root, nil
        }
    }

    // Priority 3: Walk up from cwd looking for indicator files
    cwd, _ := os.Getwd()
    for dir := cwd; dir != "/"; dir = filepath.Dir(dir) {
        for _, indicator := range []string{"go.mod", "package.json", "pyproject.toml", "DESCRIPTION"} {
            if _, err := os.Stat(filepath.Join(dir, indicator)); err == nil {
                return dir, nil
            }
        }
    }

    return "", fmt.Errorf("cannot detect project root: set GOGENT_PROJECT_DIR or run from a git repository")
}
```

**Validation**: After detection, warn if no `.claude/` directory exists in detected root.

The Go binary reads `project_root` from config.json — it does NOT detect it at runtime. Detection happens at planning time only.

#### 2. Wave Computation from blocked_by Relationships

A Go helper binary `gogent-compute-waves` processes the DAG:

```bash
gogent-compute-waves .claude/tmp/task-dag.json > .claude/tmp/waves.json
```

**Algorithm:**
1. Wave 0: All tasks with `blocked_by == []`
2. Wave 1: All tasks with `blocked_by` only referencing Wave 0 tasks
3. Wave N: All tasks with `blocked_by` only referencing Wave 0..N-1 tasks
4. Detect cycles, error if found

**waves.json structure:**

```json
{
  "waves": [
    {
      "index": 0,
      "members": ["TC-001"]
    },
    {
      "index": 1,
      "members": ["TC-002", "TC-003"]
    }
  ]
}
```

##### Wave Computation Algorithm (Review Item 18)

Uses Kahn's algorithm for topological sort with cycle detection:

```go
type Task struct {
    ID        string
    BlockedBy []string
}

func computeWaves(tasks []Task) ([][]string, error) {
    inDegree := make(map[string]int)
    dependents := make(map[string][]string)

    for _, t := range tasks {
        if _, ok := inDegree[t.ID]; !ok {
            inDegree[t.ID] = 0
        }
        for _, dep := range t.BlockedBy {
            dependents[dep] = append(dependents[dep], t.ID)
            inDegree[t.ID]++
        }
    }

    var waves [][]string
    for len(inDegree) > 0 {
        var wave []string
        for id, deg := range inDegree {
            if deg == 0 {
                wave = append(wave, id)
            }
        }

        if len(wave) == 0 {
            var remaining []string
            for id := range inDegree {
                remaining = append(remaining, id)
            }
            return nil, fmt.Errorf("circular dependency detected among: %v", remaining)
        }

        sort.Strings(wave)
        waves = append(waves, wave)

        for _, id := range wave {
            delete(inDegree, id)
            for _, dependent := range dependents[id] {
                if _, ok := inDegree[dependent]; ok {
                    inDegree[dependent]--
                }
            }
        }
    }

    return waves, nil
}
```

**Missing dependency behavior**: Error if a dependency is not in the task list (strict mode). This catches typos in `blocked_by` fields early.

**Edge case test expectations:**

| Input | Expected Output |
|-------|----------------|
| A→B→C | [[A], [B], [C]] |
| A, B (no deps) | [[A, B]] |
| A→C, B→C, C→D | [[A, B], [C], [D]] |
| A→B, B→A | Error: "circular dependency" |
| A→X (X not in list) | Error: "unknown dependency X" |

## Code Guidelines

### Prompt Engineering Patterns

**Structured prompts for clarity:**
- Use numbered sections (1. TASK, 2. CONTEXT, 3. EXPECTED OUTPUT)
- Separate instructions from data (config.json fields vs. free-form context)
- Explicit schema references (`$schema` field in all JSON)

**Freeform prompts for agent orientation:**
- Problem Brief (braintrust): Narrative format, user's words preserved
- Task descriptions (impl-manager): Plain English with clear acceptance criteria

**Template variable resolution strategy:**
- Write literal `$TEAM_DIR`, `$PROJECT_ROOT` to config.json
- Binary resolves at runtime (not at config generation time)
- Document which variables are supported in each template

### Absolute Path Handling

**Critical rules:**
1. ALL paths in stdin files must be absolute
2. Resolve at config generation time using `realpath` or `pwd` + relative path concatenation
3. Validate existence before writing (`test -f $path || error`)
4. Document in stdin schema: "All file paths MUST be absolute"

**Common path types:**

| Path Type | Resolution Strategy | Example |
|-----------|-------------------|---------|
| Project root | `pwd` in router | `/home/user/Documents/GOgent-Fortress` |
| Session files | `$team_dir` (absolute) | `/home/user/.claude/sessions/20260206.a3f2/teams/...` |
| User-provided | `realpath` during interview | `/home/user/Documents/analysis.md` |
| State files | Hardcoded absolute | `/home/user/Documents/GOgent-Fortress/.claude/tmp/specs.md` |

**Error handling:**
- If path resolution fails, error immediately (don't write invalid config)
- Include failed path in error message
- Suggest fix (e.g., "File not found: /path/to/file. Check if it exists.")

### Config.json Validation Before Launch

**Pre-flight checks (router or helper binary):**

```bash
# Required top-level fields
jq -e '.session_id' config.json > /dev/null || error "Missing session_id"
jq -e '.project_root' config.json > /dev/null || error "Missing project_root"
jq -e '.members | length > 0' config.json > /dev/null || error "No members defined"

# Validate project_root exists
project_root=$(jq -r '.project_root' config.json)
test -d "$project_root" || error "project_root does not exist: $project_root"

# Validate all stdin files exist
for member in $(jq -r '.members[].stdin_file' config.json); do
  test -f "$team_dir/$member" || error "Missing stdin file: $member"
done

# Validate wave structure
jq -e '.waves | length > 0' config.json > /dev/null || error "No waves defined"
```

**Who validates?**
- Router (bash): Basic checks before launch
- `gogent-team-run` binary: Full schema validation on startup (per TC-008)

**Validation failure behavior:**
- Pre-launch: Fail fast, display error, don't launch
- Post-launch (binary): Write error to runner.log, set status=failed in config.json, exit

## Notes

This is the most complex ticket in the plan because it integrates everything built in Phases 1-3 and fundamentally changes how users interact with multi-agent orchestration. The success of this ticket determines whether the entire team pattern delivers value.

**Risk mitigation:**
- Start with `/review` (simplest: no interview, fixed member count)
- Then `/braintrust` (interview adds complexity, but still fixed structure)
- Finally `/ticket` (most complex: dynamic members, variable waves)

**Testing strategy:**
- Manual testing first (real usage during development)
- Automated e2e tests second (after manual verification)
- Regression prevention: All 3 workflows must work after each change

**Rollback plan:**
- If team pattern doesn't work, fall back to current spawn_agent pattern
- Feature flag in settings.json: `"use_team_pattern": true/false`
- Both paths coexist until team pattern proven stable
