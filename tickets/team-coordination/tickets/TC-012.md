# TC-012: Implement Team Slash Commands

**Phase**: Phase 3: Slash Commands

**Priority**: HIGH

**Blocked By**: TC-008 (requires gogent-team-run binary and team config schema)

**Effort**: 2-3 days

## Description

Implement four slash commands for team management: `/team-status`, `/team-result`, `/team-cancel`, and `/teams`. These commands provide visibility into running and completed background orchestration teams, enable result retrieval, and support graceful cancellation.

All four commands must support the three workflow templates: braintrust, review, and implementation.

### ASCII Fallback (Review Item 25)

Auto-detect: `if os.Getenv("TERM") == "dumb" || os.Getenv("GOGENT_ASCII") == "1" { useASCII = true }`

| Unicode | ASCII | Meaning |
|---------|-------|---------|
| âś“ | `[OK]` | Completed |
| âŹł | `[..]` | Running |
| âŹ¸ | `[--]` | Pending |
| âś— | `[XX]` | Failed |
| đź”„ | `[>>]` | Retrying |

**How to Detect Running vs Completed**:
1. Check `background_pid`: if null, team is stopped
2. If `background_pid` is set, check if process exists: `kill -0 $PID 2>/dev/null`
3. If process is dead but PID is set, status is "stale" (crashed or killed)
4. Check heartbeat age: if > 60s, process likely hung

**Heartbeat Freshness Check**:
```bash
# Heartbeat file is touched every 30s by gogent-team-run
HEARTBEAT_AGE=$(($(date +%s) - $(stat -c %Y heartbeat)))
if [ $HEARTBEAT_AGE -gt 60 ]; then
  echo "âš ď¸Ź  Warning: heartbeat stale (${HEARTBEAT_AGE}s old)"
fi
```

### Heartbeat Remediation Policy (Review Item 12)

Heartbeat interval reduced from 30s to **10s** (change in `cmd/gogent-team-run/heartbeat.go`).

| Heartbeat Age | Status Display | User Action |
|--------------|---------------|-------------|
| < 30s | `heartbeat: Xs ago` (normal) | None |
| 30-60s | `âš ď¸Ź heartbeat: Xs ago (delayed)` | Check system load |
| 60-120s | `âš ď¸Ź heartbeat: Xs ago (stale â€” process may be hung)` | Consider `/team-cancel` |
| > 120s | `â›” heartbeat: Xs ago (likely dead)` | `/team-cancel` + check `runner.log` |

> **Important**: Heartbeat freshness indicates the main `gogent-team-run` process is alive, NOT that individual agents are making progress. An agent may be legitimately thinking for 5+ minutes while the heartbeat remains fresh.

### Command 2: `/team-result <team-name>`

**Purpose**: Display the final output from a completed team.

**Braintrust Workflow**:
- Read `stdout_beethoven.json` (final-wave agent)
- Extract envelope fields: `content.executive_summary`, `content.convergence_points`, `content.divergence_resolution`, `content.unified_recommendations`
- Format as markdown sections

**Review Workflow**:
- Read all reviewer stdout files: `stdout_backend-reviewer.json`, `stdout_frontend-reviewer.json`, `stdout_standards-reviewer.json`, `stdout_architecture-reviewer.json`
- Extract `content.findings[]` from each
- Aggregate by severity: CRITICAL, HIGH, MEDIUM, LOW
- Group by category within each severity level

**Implementation Workflow**:
- Read stdout files for all workers in final wave
- Extract `content.deliverables[]` from each
- Group by phase
- List modified files

**Output Format for Braintrust**:

```markdown
# Braintrust Analysis: [team_name]

## Executive Summary
[content.executive_summary from beethoven]

## Convergence Points
[content.convergence_points from beethoven]

## Divergence Resolution
[content.divergence_resolution from beethoven]

## Unified Recommendations
[content.unified_recommendations from beethoven]

---
Cost: $X.XX | Duration: Xm Xs | Agents: N
```

**Output Format for Review**:

```markdown
# Code Review: [team_name]

## CRITICAL (N findings)
### [category]
- [file:line] [description] (backend-reviewer)

## HIGH (N findings)
### [category]
- [file:line] [description] (frontend-reviewer)

[... MEDIUM, LOW ...]

---
Cost: $X.XX | Duration: Xm Xs | Reviewers: N
```

### Review Conflict Resolution (Review Item 26)

When `/team-result` aggregates review findings and multiple reviewers report the same issue at different severities:

1. Use the **highest** severity reported
2. Show attribution: `[CRITICAL] (reported by: backend, architecture)`
3. If severities differ by 2+ levels, add note: `âš ď¸Ź severity disputed â€” see individual reviews`

Example output:
```
### CRITICAL (3 findings)

1. PID File Race Condition
   Reported by: backend-reviewer, architecture-reviewer
   ...

2. Budget Gate Race Condition
   Reported by: backend-reviewer, architecture-reviewer
   âš ď¸Ź severity disputed (backend: HIGH, architecture: CRITICAL)
   ...
```

**Graceful Degradation**:
- If stdout file missing: "Agent [name] did not produce output (status: [status])"
- If stdout malformed: Log raw content to stderr, display fallback message
- If required fields missing from envelope: "Output incomplete - missing [field]"

### Command 3: `/team-cancel <team-name>`

**Purpose**: Gracefully stop a running team.

**Implementation Steps**:

1. **Read config.json** for `background_pid`
2. **Verify process exists**: `kill -0 $PID 2>/dev/null`
3. **Send SIGTERM**: `kill -TERM $PID`
4. **Wait for graceful shutdown** (up to 10 seconds)
5. **Verify termination**: Check if process still exists
6. **Update config.json**: Set `orchestrator.status = "cancelled"`, `orchestrator.completed_at = now()`
7. **Display confirmation**

**SIGTERM Cascade**:
- `gogent-team-run` catches SIGTERM
- Sends SIGTERM to all child `claude` processes
- Waits up to 5s for children to exit
- Sends SIGKILL to any remaining children
- Cleans up PID file
- Exits

**Error Cases**:
- PID is null: "Team is not running"
- Process doesn't exist: "Team process not found (PID ${PID} is stale)"
- Process won't die after 10s: "Warning: Team did not stop gracefully. Force kill? (y/n)"

**Force Kill Option**:
- If user confirms, send SIGKILL: `kill -9 $PID`
- Update config.json as "killed" rather than "cancelled"

**Output**:

```
Cancelling team: braintrust-1738856422 (PID 12345)
Sent SIGTERM...
Waiting for graceful shutdown...
Team stopped successfully.

Final status:
- Wave 1: COMPLETED (2/2)
- Wave 2: CANCELLED (0/1, beethoven was running)
- Cost: $2.86 / $15.00
- Duration: 3m 12s
```

## Code Guidelines

### Implementation Mechanism: Claude Code Skills

After investigating existing patterns, these commands should be **Claude Code skills** (not Go binaries). Rationale:

**Skills Pattern Observed**:
- `/braintrust`, `/explore`, `/review` are all skills (YAML + prompt)
- Skills can use Bash, Read, Glob tools within the prompt
- Skills provide rich LLM-driven output formatting (markdown, tables)
- Skills can make decisions (e.g., graceful degradation on missing files)

**Go Binary Pattern Observed**:
- `gogent-archive`, `gogent-scout`, `gogent-validate` are pure automation
- Deterministic input â†’ deterministic output
- No conditional logic based on file content interpretation
- Performance-critical or run as hooks

**Team Commands Characteristics**:
- Need to interpret JSON content (executive summary extraction)
- Require flexible markdown formatting
- Handle graceful degradation (missing files, malformed JSON)
- User-facing output quality matters more than speed
- **Conclusion**: Skills are the better fit

## Design Decision: Session Directory Discovery

**Problem**: Slash commands need to find the current session's team directories. Three options exist.

### Option A: GOGENT_SESSION_DIR Environment Variable

**Mechanism**: TUI sets `export GOGENT_SESSION_DIR=/path/to/sessions/260206.abc123` before spawning Claude CLI.

**Pros**:
- Explicit, no filesystem scanning
- Works across nested subshells
- Fast (no find/glob operations)

**Cons**:
- Requires TUI modification to set the env var
- Env var might not propagate to skills in all contexts
- Fragile if TUI architecture changes

**Implementation Effort**: 0.5 days (TUI change + skill reading env var)

---

### Option B: Filesystem Scan

**Mechanism**: Skills scan `~/.claude/sessions/*/teams/` for directories with `config.json`, filter by timestamp (most recent N hours).

**Pros**:
- No TUI changes required
- Works immediately
- Resilient to session migration

**Cons**:
- Slow on large session histories (100+ sessions)
- Ambiguous if multiple sessions active (rare but possible)
- Requires heuristic to identify "current" session

**Implementation Effort**: 0.5 days (glob + filter logic in skill)

---

### Option C: TUI State Query via MCP

**Mechanism**: MCP tool `mcp__ide__getSessionInfo()` returns current session directory.

**Pros**:
- Clean separation of concerns
- TUI owns session state, skills query it
- No env var propagation concerns

**Cons**:
- Requires new MCP tool implementation
- Higher complexity (TypeScript + protocol)
- Skills become dependent on TUI MCP server

**Implementation Effort**: 1-2 days (MCP tool + skill integration)

---

### Recommendation: Option A (Environment Variable)

**Rationale**:
1. **Fastest to implement** (0.5 days vs 1-2 days for MCP)
2. **Explicit and deterministic** (no filesystem heuristics)
3. **Matches existing pattern**: TUI already sets env vars like `CLAUDE_CODE_EFFORT_LEVEL` for spawned agents
4. **Easy to debug**: `echo $GOGENT_SESSION_DIR` shows the value

**Implementation**:
- TUI: In `packages/tui/src/store/slices/session.ts`, set `process.env.GOGENT_SESSION_DIR = sessionDir` during session init
- Skills: Read `process.env.GOGENT_SESSION_DIR` or fallback to filesystem scan if unset

**Fallback Strategy**:
If `GOGENT_SESSION_DIR` is unset (e.g., running skills outside TUI), fall back to Option B:
```bash
# Find most recent session with teams
SESSION_DIR=$(find ~/.claude/sessions -maxdepth 2 -name teams -type d | \
  xargs -I {} dirname {} | \
  xargs ls -dt | head -1)
```

## Notes

### Concurrent Teams

The design supports **concurrent teams** (e.g., braintrust + review running simultaneously):
- Each team has its own directory
- Each team has its own PID
- `/teams` lists all
- `/team-status <name>` disambiguates

### Session Migration

If the user copies a session directory to a different machine:
- Env var approach breaks (different session ID)
- Filesystem scan fallback works
- PIDs become stale (handled gracefully)

### Cost Tracking

Team commands display cost from `config.json`, which is updated by `gogent-team-run` after each agent completes. The source of truth for cost is the `claude` CLI JSON output, extracted by `gogent-team-run` and written to config.

### Output Redirection

Skills output to stdout. Users can redirect:
```bash
/team-result braintrust-123 > analysis.md
```

This enables saving results to files for later reference or sharing.
