# TC-015a: Input Buffer (Message Queuing During Streaming)

**Phase**: Independent
**Priority**: P0 (HIGH)
**Blocked By**: none
**Effort**: 4-6 hours
**Status**: pending

---

## Description

Decouple user input from the streaming guard in `useClaudeQuery.ts`. Currently, line 600's `if (streamingRef.current) { return; }` silently drops all user messages while any foreground query is streaming. Additionally, `ClaudePanel.tsx` disables the input field entirely during streaming (line 307 `disabled={streaming}`, line 246 keymap disable, line 146 handleSubmit guard).

**Target behavior:** User can type and submit messages while a query is streaming. Messages queue and auto-send when the current query completes. No concurrent queries needed — just a simple FIFO buffer.

**Why this matters:** With team-run handling orchestration (TC-013a/b/c), the remaining TUI freeze is 5-30 second blocks during normal foreground queries. This captures 90% of the concurrent query value at 5% of the cost by letting users queue their next message instead of waiting.

---

## Root Cause (3 blocking mechanisms)

### 1. sendMessage guard (useClaudeQuery.ts:600)

```typescript
if (streamingRef.current) {
  void logger.warn("Query already in progress, ignoring duplicate call");
  return;
}
```

Messages are silently dropped.

### 2. TextInput disabled prop (ClaudePanel.tsx:307)

```typescript
<TextInput disabled={streaming} ... />
```

The input field is visually and functionally disabled.

### 3. Keymap disabled (ClaudePanel.tsx:246)

```typescript
useKeymap(panelBindings, focused && modalQueue.length === 0 && !streaming);
```

Submit keybinding is disabled entirely during streaming.

## Implementation Instructions

### Step 1: Enable input during streaming

**File:** `packages/tui/src/components/ClaudePanel.tsx`

- Remove `disabled={streaming}` from `TextInput` (line 307)
- Remove `&& !streaming` from keymap enable condition (line 246)
- In `handleSubmit` (line 146), replace the `if (!trimmedInput || streaming) return;` guard:

```typescript
// BEFORE
if (!trimmedInput || streaming) return;

// AFTER
if (!trimmedInput) return;
if (streaming) {
  // Queue for later instead of dropping
  enqueueMessage(trimmedInput);
  setInput("");
  return;
}
```

### Step 2: Add message queue state

**Option A (Local state — simpler, recommended for MVP):**

Add to `ClaudePanel.tsx`:

```typescript
const [pendingMessage, setPendingMessage] = useState<string | null>(null);

const enqueueMessage = useCallback((message: string) => {
  setPendingMessage(message);
  // Visual feedback
  addSystemMessage(`Message queued — will send when current response completes.`);
}, [addSystemMessage]);
```

**Option B (Zustand store — if queue needs to be visible elsewhere):**

Add to `UISlice` in `packages/tui/src/store/types.ts`:

```typescript
pendingMessage: string | null;
setPendingMessage: (msg: string | null) => void;
```

### Step 3: Auto-drain queue after streaming completes

**File:** `packages/tui/src/hooks/useClaudeQuery.ts`

In `handleResultEvent` (around line 575-578), after streaming stops, check for queued message:

```typescript
// Existing streaming-off logic
setIsStreaming(false);
streamingRef.current = false;
setStreamingState(false);

// NEW: drain queued message
const pending = getPendingMessage(); // from store or callback
if (pending) {
  clearPendingMessage();
  // Small delay to let React render the completed response
  setTimeout(() => void sendMessage(pending), 100);
}
```

The `getPendingMessage` / `clearPendingMessage` access pattern depends on whether state is local (callback prop) or in Zustand (store access).

### Step 4: Visual feedback

Show the queued message state in the input area or status line:

```typescript
// In ClaudePanel, below the TextInput:
{pendingMessage && (
  <Text dimColor>Queued: {pendingMessage.slice(0, 60)}{pendingMessage.length > 60 ? "..." : ""}</Text>
)}
```

## Success Criteria

## Design Decisions

### Single message queue, not multi-message

MVP supports one pending message (last-write-wins if user submits twice). Multi-message queue adds complexity (ordering, cancellation UI) with minimal practical value — users rarely queue more than one.

### No concurrent queries

This ticket explicitly avoids the concurrent query refactor (TC-015c). The streaming guard in `sendMessage` (line 600) is changed from "drop" to "enqueue", but only one query runs at a time. The queue drains sequentially.

### Local state vs Zustand

Recommend local state (Option A) unless another component needs to read the queue. The input buffer is entirely within ClaudePanel's responsibility.
