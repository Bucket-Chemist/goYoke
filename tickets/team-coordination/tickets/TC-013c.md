## Description

Migrate the `/ticket` implementation workflow from foreground sequential agent spawning to background `gogent-team-run` execution with computed wave DAGs.

**Current behavior:** Router invokes impl-manager via `Task(sonnet)`, which reads specs.md, builds a task DAG, and spawns worker agents (go-pro, python-pro, etc.) sequentially via `Task()`. TUI freezes for 5-15 minutes.

**Target behavior:** Router reads specs.md, invokes `gogent-plan-impl` (Go binary) to parse tasks and compute waves via Kahn's algorithm, generates config.json + stdin files, launches `gogent-team-run`. TUI returns in <15 seconds.

**Key challenges:**
- Dynamic member count (variable number of tasks from specs.md)
- Variable wave count (depends on dependency graph)
- Requires new Go binary `gogent-plan-impl` (consolidated from TC-013's 4 proposed binaries)
- specs.md format must be stable enough for Go parsing

**Reference:** `tickets/team-coordination/ImplMgr-team-bridge.md` (authoritative corrections)

## Implementation Instructions

### 1. Build `gogent-plan-impl` Go Binary

Single binary that replaces TC-013's 4 proposed binaries (`gogent-team-init-impl`, `gogent-parse-specs`, `gogent-compute-waves`, plus config generation).

```
cmd/gogent-plan-impl/main.go

Usage: gogent-plan-impl --specs=<path> --project-root=<path> --output=<team-dir>

The binary:
1. Reads and parses specs.md to extract tasks
2. Computes wave DAG via Kahn's algorithm (topological sort)
3. Generates config.json from schemas/teams/implementation.json template
4. Generates stdin files per worker (compliant with schemas/stdin/worker.json)
5. Writes all files to --output directory
```

### 2. specs.md Parsing

Expected specs.md format (from architect agent output):
```markdown
## Tasks

### task-001: Implement authentication handler
- **Agent**: go-pro
- **Packages**: internal/handlers
- **Files**: internal/handlers/auth.go, internal/handlers/auth_test.go
- **Blocked By**: (none)
- **Acceptance Criteria**:
  - Handler validates JWT tokens
  - Returns 401 for expired tokens
  - Table-driven tests

### task-002: Write integration tests
- **Agent**: go-pro
- **Packages**: internal/handlers
- **Files**: internal/handlers/integration_test.go
- **Blocked By**: task-001
- **Acceptance Criteria**:
  - End-to-end auth flow tested
```

Parser extracts: task_id, subject, agent, packages, files, blocked_by, acceptance_criteria.

### 3. Wave Computation (Kahn's Algorithm)

```
Input: tasks with blocked_by relationships
Output: waves where wave N+1 only contains tasks whose dependencies are all in waves 0..N

Algorithm:
1. Compute in-degree for each task
2. Wave 0: all tasks with in-degree 0
3. Remove Wave 0, decrement dependents' in-degrees
4. Wave 1: all tasks with in-degree 0
5. Repeat until empty
6. If tasks remain with in-degree > 0: circular dependency error
```

Edge cases:
- A->B->C: [[A], [B], [C]]
- A, B (no deps): [[A, B]]
- A->C, B->C, C->D: [[A, B], [C], [D]]
- A->B, B->A: Error: "circular dependency among: A, B"
- A->X (X not in list): Error: "unknown dependency X referenced by A"

### 4. Stdin File Generation

Each task gets a stdin file compliant with `schemas/stdin/worker.json`. Required fields:
- `agent`: dynamic (from specs.md task agent field)
- `workflow`: `"implementation"` (const)
- `context`: `{project_root, team_dir}` (absolute paths)
- `task`: `{task_id, subject, description, acceptance_criteria, blocked_by, blocks}` — NOTE: this is an **object**, not a string. The envelope builder uses the separate `description` field (string) for validation.
- `implementation_scope`: `{target_packages, related_files, tests_required}`
- `conventions`: `{language, conventions_file}` (inferred from agent type)
- `codebase_context`: `{architecture_notes, patterns_to_follow, anti_patterns}`
- `description`: string — **required by envelope builder**. Use format: `"Implement: {task.subject}"`

See `ImplMgr-team-bridge.md` Section 4 for complete validated example.

### 5. Config.json Generation

From `schemas/teams/implementation.json` template:
- `waves[]`: computed by Kahn's algorithm
- `waves[].members[]`: tasks assigned to each wave
- `wave_number`: 1-indexed (per template convention)
- `on_complete_script`: null for implementation waves

### 6. Launch Sequence

```bash
# 1. Parse and generate
gogent-plan-impl \
  --specs=".claude/tmp/specs.md" \
  --project-root="$(pwd)" \
  --output="$team_dir"

# 2. Launch
gogent-team-run "$team_dir"

# 3. Verify
sleep 2
background_pid=$(jq -r '.background_pid' "$team_dir/config.json")
# ... standard verification
```

### 7. Fallback for Simple Tickets

If ticket has no specs.md, fall back to current behavior (direct delegation to language agent, no team pattern). The team pattern is only used when specs.md exists with multiple tasks.

### 8. Feature Flag

Same as TC-013a/b: `settings.json -> "use_team_pattern": true/false`.

## Test Cases

### Test 1: Multi-wave DAG (5 tasks, 3 waves)
- specs.md: 5 tasks with dependency chain
- Wave 1: 2 tasks (no deps), Wave 2: 1 task, Wave 3: 2 tasks
- All complete successfully

### Test 2: Circular dependency detection
- specs.md: task-001 blocked_by task-002, task-002 blocked_by task-001
- `gogent-plan-impl` errors: "circular dependency detected"
- No team launched

### Test 3: Single-task (no specs.md)
- Ticket has no specs.md
- Falls back to direct delegation (no team pattern)

### Test 4: Task failure mid-wave (requires wave.go modification)
- Wave 1 has 2 tasks, one fails after retries
- **Expected behavior:** After Wave 1 completes, `runWaves` checks member statuses. If any member has `status: failed`, subsequent waves are skipped and their members are set to `status: skipped`.
- Config.json shows: Wave 1 member with `status: failed`, Wave 2 members with `status: skipped`
- **Implementation:** Modify `runWaves()` in `wave.go` to check member statuses after `wg.Wait()` and before proceeding to next wave. Currently `runWaves` iterates unconditionally.
- **Test file:** Add test to `wave_test.go` that spawns a wave with a deliberately failing member and verifies Wave 2 members get `status: skipped`

### Test 5: Unknown dependency
- specs.md references non-existent task in blocked_by
- `gogent-plan-impl` errors: "unknown dependency"

## Notes

This is the most complex of the three decomposed tickets. The `gogent-plan-impl` binary is the main deliverable — it consolidates TC-013's originally proposed 4 binaries into 1.

The specs.md format is not formally specified. The parser should be tolerant of formatting variations (extra whitespace, missing optional fields, etc.) while strict on required fields (task_id, agent, blocked_by).

Consider: should `gogent-plan-impl` validate that referenced agents exist in `agents-index.json`? Recommended: yes, fail early with clear error.
