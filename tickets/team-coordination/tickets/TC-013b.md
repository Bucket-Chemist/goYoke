## Description

Migrate the `/braintrust` workflow from foreground MCP-spawn orchestration to a hybrid foreground-interview + background-execution pattern using `gogent-team-run`.

**Current behavior:** Mozart conducts interview (foreground), then spawns Einstein + Staff-Architect + Beethoven via `mcp__gofortress__spawn_agent` (foreground, ~6-9 minutes TUI freeze).

**Target behavior:** Mozart conducts interview (foreground, ~30s), generates team config + stdin files, launches `gogent-team-run` (background), returns to user immediately. Background execution: Wave 1 (Einstein + Staff-Architect parallel) -> inter-wave synthesis (`gogent-team-prepare-synthesis`) -> Wave 2 (Beethoven synthesis).

**Key challenge:** Mozart's `mozart.md` Phase 2.5 stdin templates (from TC-018) do not comply with TC-009's `einstein.json` schema. These must be reconciled.

**Reference:**
- `tickets/team-coordination/team-dispatch-decision-matrix.md` (dispatch patterns)
- `.claude/schemas/stdin/einstein.json` (authoritative stdin structure)
- `.claude/schemas/teams/braintrust.json` (team template)

## Implementation Instructions

### 1. Update Mozart Phase 2.5 Stdin Templates

`~/.claude/agents/mozart/mozart.md` Phase 2.5 (section "Stdin Template Generation") has stdin templates that predate TC-009 schemas. Replace with schema-compliant templates.

> **Note:** Reference by section header, not line numbers. Line numbers shift across edits.

**Current (TC-018, STALE):**
```json
{
  "task": {"problem_statement": "...", "type": "theoretical_analysis"},
  "context": {"relevant_files": [...]},
  "reads_from": {"scout_metrics": "..."},
  "writes_to": {"output": "wave1/einstein-analysis.md"},
  "paths": {"project_root": "..."}
```

**Required (schemas/stdin/einstein.json):**
```json
{
  "agent": "einstein",
  "workflow": "braintrust",
  "context": {"project_root": "/abs/path", "team_dir": "/abs/path"},
  "problem_brief": {
    "title": "...",
    "statement": "...",
    "scope": {"in_scope": [], "out_of_scope": [], "affected_modules": []},
    "complexity_signals": {"cross_module": false, "novel_problem": true},
    "constraints": {"technical": [], "organizational": [], "compatibility": []}

    "architecture_summary": "...",

  },
  "scout_findings": {"metrics": {}, "summary": "..."}
}
```

Similar updates needed for `stdin_staff-architect.json` and `stdin_beethoven.json` templates.

**All stdin templates must include a `description` field** (string) for envelope builder compatibility:
- Einstein: `"Theoretical analysis: {problem_brief.title}"`
- Staff-Architect: `"Critical review: {problem_brief.title}"`
- Beethoven: `"Synthesize analyses for: {problem_brief.title}"`

### 2. Add Phase 6 Team-Run Dispatch to mozart.md

Mozart Phases 6-7 currently describe MCP spawn pattern. Add a parallel section for team-run dispatch:

**After interview confirmation (Phase 5):**
1. Create team directory under session
2. Write Problem Brief to team directory
3. Generate config.json from `schemas/teams/braintrust.json` template
4. Generate 3 stdin files (einstein, staff-architect, beethoven)
5. Launch `gogent-team-run "$team_dir"`
6. Verify launch (read `background_pid` from config.json)
7. Return to user: "Braintrust team dispatched. Use /team-status."

**Decision point:** Use `settings.json -> "use_team_pattern"` flag to choose between MCP spawn (Phase 6-7 original) and team-run (Phase 6 new).

### 3. Inter-Wave Synthesis Integration

The braintrust template sets `waves[0].on_complete_script: "gogent-team-prepare-synthesis"` for Wave 1. After Einstein + Staff-Architect complete, the Go binary calls this script which:
1. Reads `stdout_einstein.json` and `stdout_staff-architect.json`
2. Extracts key sections with graceful degradation
3. Writes `pre-synthesis.md` to team directory
4. Beethoven (Wave 2) receives this as input

**Verify:** `gogent-team-prepare-synthesis` binary is on PATH (`go install ./cmd/gogent-team-prepare-synthesis/`).

### 4. Beethoven Stdin — pre_synthesis_path Mechanism

Beethoven's stdin is written **before Wave 1 runs**, so it cannot contain Einstein/Staff-Architect output objects. Instead, Beethoven receives a file path and reads the synthesis at runtime.

**Mechanism:** The `beethoven.json` schema now requires `pre_synthesis_path` (absolute path string) instead of `einstein_analysis`/`staff_architect_review` objects. The lifecycle:
1. Config generator writes `stdin_beethoven.json` with `pre_synthesis_path` pointing to `{team_dir}/pre-synthesis.md`
2. Wave 1 runs (Einstein + Staff-Architect produce stdout files)
3. Inter-wave script (`gogent-team-prepare-synthesis`) reads stdout files → writes `pre-synthesis.md`
4. Wave 2 starts: Beethoven reads `pre_synthesis_path` via Read tool at runtime

**stdin_beethoven.json must include:**
- `pre_synthesis_path`: `"{team_dir}/pre-synthesis.md"` (absolute path)
- `problem_brief`: copy of the problem brief from Mozart
- `description`: `"Synthesize analyses for: {problem_brief.title}"`

### 5. Budget

**Resolved:** Budget raised to **$16.00** in `braintrust.json` template (Phase 0 fix D-2).

**Arithmetic:** `estimateCost("opus")` returns $5.00 (config.go:379). With 3 Opus agents (Einstein, Staff-Architect, Beethoven), the minimum reservation is $15.00. Budget of $16.00 provides $1.00 margin. Budget is a reservation ceiling — actual costs are reconciled to real usage after each agent completes via `reconcileCost()`.

**Warning threshold:** Set to $13.00 (80% of $16.00) for early visibility.

### 6. Feature Flag

Same as TC-013a: `settings.json -> "use_team_pattern": true/false`. When false, Mozart uses existing MCP spawn path (Phases 6-7 original).
