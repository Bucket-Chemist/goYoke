# TC-015: TUI Concurrent Query Support

**Phase**: Independent Tickets

**Priority**: MEDIUM

**Blocked By**: none (TC-019 Phase 1 gate cleared 2026-02-07)

**Effort**: 3-5 days

## Description

Fix the root cause of TUI freezing during agent orchestration: the `useClaudeQuery` hook treats the streaming state as a global mutex, preventing user interaction during any active query.

## Root Cause Analysis

### Actual Code Behavior (from `useClaudeQuery.ts`)

**The Freeze Mechanism:**

1. **Line 152-153**: `isStreaming` is component state, `streamingRef` is a ref that tracks the same value
2. **Line 600-603**: The guard `if (streamingRef.current)` prevents concurrent `sendMessage()` calls:
   ```typescript
   if (streamingRef.current) {
     void logger.warn("Query already in progress, ignoring duplicate call");
     return;
   }
   ```
3. **Lines 629-631**: Streaming starts by setting BOTH state and ref:
   ```typescript
   setIsStreaming(true);
   streamingRef.current = true;
   setStreamingState(true);  // Updates Zustand store
   ```
4. **Lines 730-750**: The `for await (const event of eventStream)` loop runs to completion
5. **Lines 575-578** (in `handleResultEvent`): Streaming ends only when result event arrives:
   ```typescript
   setIsStreaming(false);
   streamingRef.current = false;
   setStreamingState(false);
   ```

**Critical insight**: The freeze is NOT in the Node.js event loop. The event loop continues processing I/O, timers, and Ink renders. The freeze is **application-level**: `sendMessage()` has an early return guard that blocks all new user messages while `streamingRef.current === true`.

**Why this blocks orchestration:**

When Mozart spawns Einstein via `mcp__gofortress__spawn_agent`:
1. Mozart's query is in the "streaming" state
2. The `spawn_agent` MCP tool is invoked as part of the event stream
3. Einstein's CLI process runs (potentially 30+ seconds)
4. The `for await` loop blocks waiting for Einstein's completion event
5. During this time, `streamingRef.current === true`
6. User cannot send new messages (blocked by line 600 guard)
7. User cannot cancel (no UI for it; Ctrl+C kills entire TUI)

### Phase 2: Refactor State Management (if SDK allows concurrency)

#### 2.1 Current State (Single-Query Model)

**Global streaming state (lines 152-159, 629-631, 575-578)**:
- `isStreaming`: React state (boolean)
- `streamingRef`: Ref tracking same value (avoids stale closures)
- `setStreamingState(...)`: Updates Zustand store `UISlice.streaming`

**Guard mechanism (lines 600-603)**:
- Blocks all `sendMessage()` calls while any query is active
- Uses ref instead of state to avoid stale closure issues

**Event stream reference (line 182, 721)**:
- `eventStreamRef`: Stores current query for `setModel()` and `streamInput()` calls
- Single reference = only one active query at a time

#### 2.2 Target State (Multi-Query Model)

**Per-query streaming state**:
```typescript
// New type for query lifecycle
interface ActiveQuery {
  id: string;                    // Unique query ID (nanoid)
  eventStream: Awaited<ReturnType<typeof query>>;
  isStreaming: boolean;
  startedAt: number;
  currentMessageId: string | null;
  currentMessageRef: ContentBlock[];
}

// New state in useClaudeQuery
const [activeQueries, setActiveQueries] = useState<Map<string, ActiveQuery>>(new Map());
```

**Concurrency policy**:
- Multiple queries can run simultaneously
- Each query has independent streaming state
- User input is NOT blocked during streaming
- Each query maintains its own message accumulation buffer

**Store updates (Zustand)**:
- `UISlice.streaming` becomes `UISlice.activeQueryCount: number`
- UI shows "X queries active" instead of binary streaming state

**Step 2: Create per-query state container**

```typescript
const sendMessage = useCallback(async (message: string): Promise<void> => {
  const queryId = nanoid();

  const newQuery: ActiveQuery = {
    id: queryId,
    eventStream: null as any, // Will be set after query() call
    isStreaming: true,
    startedAt: Date.now(),
    currentMessageId: null,
    currentMessageRef: [],
  };

  // Add to active queries
  setActiveQueries(prev => new Map(prev).set(queryId, newQuery));

  // Update global streaming count
  setStreamingState(true); // Or increment counter

  try {
    // ... existing query setup ...

    const eventStream = query({ /* ... */ });

    // Update query with stream reference
    setActiveQueries(prev => {
      const updated = new Map(prev);
      const q = updated.get(queryId);
      if (q) q.eventStream = eventStream;
      return updated;
    });

    // Event loop with query-scoped state
    for await (const event of eventStream) {
      // Update THIS query's message buffer (not global)
      // Pass queryId to event handlers
    }
  } finally {
    // Remove from active queries
    setActiveQueries(prev => {
      const updated = new Map(prev);
      updated.delete(queryId);
      return updated;
    });

    // Update global streaming count
    if (activeQueries.size === 0) {
      setStreamingState(false);
    }
  }
}, []);
```

## Code Guidelines

### TypeScript Strict Mode

- No `any` types - use proper `ActiveQuery` interface
- All Promise returns must be awaited or voided
- Event handler signatures must match SDK types exactly

### React Hooks Rules

**Dependencies**:
- `useCallback` deps must include all referenced state/props
- `activeQueries` Map updates require new Map instance (immutability)
- `useEffect` cleanup must remove query from active set

**State Updates**:
```typescript
// CORRECT: Immutable Map update
setActiveQueries(prev => new Map(prev).set(queryId, newQuery));

// WRONG: Mutating Map
setActiveQueries(prev => {
  prev.set(queryId, newQuery);  // Mutation!
  return prev;
});
```

### Zustand Store Patterns

- Use `get()` for cross-slice reads inside actions
- Selectors with `shallow` comparison for derived state
- No direct mutation of nested objects

### Concurrency Hazards

**Race condition**: Query completes before `setActiveQueries` runs
```typescript
// MITIGATION: Use finally block for cleanup
try {
  for await (const event of eventStream) { /* ... */ }
} finally {
  setActiveQueries(prev => {
    const updated = new Map(prev);
    updated.delete(queryId);
    return updated;
  });
}
```

**Memory leak**: Failed queries not removed from active set
```typescript
// MITIGATION: finally block runs even on exception
```

## Open Questions

### Q1: Does Claude Agent SDK support concurrent `query()` calls?

**Status**: ANSWERED (TC-019 investigation, 2026-02-07)

**Answer: YES.** The SDK supports concurrent `query()` calls from the same Node.js process. Each call spawns an independent Claude CLI subprocess with isolated stdio streams.

**Evidence** (see `packages/tui/docs/sdk-concurrency-investigation.md`):
- Test 1: Two simultaneous queries both completed successfully (8.4s)
- Test 2: Second query does NOT cancel first (both complete independently)
- Test 3: Output streams fully isolated (no cross-contamination)

**SDK Version Tested**: 0.2.31 | **Node.js**: v25.4.0

**Implication**: Phase 2 can proceed as designed. The TUI freeze is an application-level guard (`streamingRef.current` check at line 598), NOT an SDK limitation.

### Q2: How should message ordering work with concurrent queries?

**Current behavior**: Messages are appended in order of completion, not order of initiation.

**Options**:
- **A**: Maintain current behavior (completion order)
- **B**: Interleave messages by timestamp
- **C**: Group messages by query (show query boundaries in UI)

**Recommendation**: Option A for MVP (simplest), Option C for UX improvement later.

### Q3: Should there be a concurrency limit?

**Consideration**: 5+ concurrent Opus queries could exceed rate limits or budget.

**Options**:
- **A**: No limit (rely on API rate limiting)
- **B**: Hard limit (e.g., max 3 concurrent queries)
- **C**: Budget-based limit (total active cost < $X)

**Recommendation**: Option A for MVP. Add limit if users report issues.

### Q4: How to handle built-in tool modals from concurrent queries?

**Current behavior**: Modal queue is FIFO across all queries.

**Issues**:
- User sees modal for Query B while Query A is waiting for tool result
- Context switching is confusing

**Options**:
- **A**: Keep FIFO queue (current behavior)
- **B**: Block new queries while modal is active
- **C**: Show modal header with query context ("Query: What is 2+2?")

**Recommendation**: Option C (add query context to modal).
