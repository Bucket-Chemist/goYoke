# TC-021: Modernize Existing Skills with Official Skill Features

**Phase**: Phase 3.5: Skill Modernization

**Priority**: MEDIUM

**Blocked By**: TC-012 (validates the new skill patterns first)

**Effort**: 2-3 days

## Description

Update all existing skills to use official Claude Code skill features identified in the skills documentation audit. Six skills need updating: `/braintrust`, `/review`, `/explore`, `/plan`, `/review-plan`, `/ticket`. Additionally, fix the reviewer JSON stdin/stdout propagation gap. Canonical JSON Schema validation files exist at `schemas/stdin/reviewer.json` and `schemas/stdout/reviewer.json` (proper `$schema`, `$id`, `required`, `additionalProperties: false`) but aren't referenced by `gogent-team-run` for validation or by the `/review` skill for stdin generation. The per-domain example files at `schemas/teams/stdin-stdout/review-*.json` provide domain-specific focus area templates but are also unused.

### Why This Matters

1. **Cost protection**: Without `disable-model-invocation: true`, Claude can auto-trigger $4-6 braintrust invocations based on description matching
2. **Reviewer JSON gap**: Schemas for backend/frontend/standards/architect reviewer stdin/stdout exist but aren't connected to the actual `/review` workflow — the team pattern solves this
3. **Isolation**: `/review` is a natural `context: fork` candidate — reviewers don't need conversation history
4. **Hook integration**: PostToolUse hooks can automate the PLAN→EXECUTE transition for team-based workflows, replacing prompt-dependent Bash calls to `gogent-team-run`

## Part 2: Update `/review` to Team Pattern with `context: fork`

This is the largest change. The current `/review` skill runs inline and orchestrates reviewers via Task() calls without structured stdin/stdout. The new version uses `context: fork` with the team pattern, which naturally solves the reviewer JSON propagation gap.

### Current Flow (Broken)

```
/review → inline orchestration → Task(backend-reviewer) → unstructured output
                                → Task(frontend-reviewer) → unstructured output
                                → Task(standards-reviewer) → unstructured output
                                → Task(architect-reviewer) → unstructured output
        → manual synthesis in conversation context
```

**Problems**:
- No structured stdin (reviewers don't receive the schema-defined input)
- No structured stdout (reviewers produce free-form text, not schema-conformant JSON)
- Findings aggregation is prompt-dependent
- Results pollute the main conversation context

### New Flow (Team Pattern)

```
/review → INLINE (NOT forked — needs to write multiple files + different models per reviewer)
        → !`command` pre-injects: git diff, changed files, languages
        → Skill generates:
            1. team config.json (from review.json template)
            2. stdin_backend-reviewer.json (from schema, haiku)
            3. stdin_frontend-reviewer.json (from schema, haiku)
            4. stdin_standards-reviewer.json (from schema, haiku)
            5. stdin_architect-reviewer.json (from schema, sonnet)
        → PostToolUse hook on Write detects config.json → launches gogent-team-run
        → gogent-team-run spawns reviewers in parallel with per-agent models
        → Reviewers produce stdout matching schemas
        → gogent-team-run writes stdout_*.json files
        → User retrieves results via /team-result (which CAN be forked — single read-only agent)
```

**Why NOT `context: fork`**: Forked skills run as a single subagent with a single model. The review workflow requires 4 reviewers with different models (haiku for backend/frontend/standards, sonnet for architect). Multi-model parallel spawning must happen via `gogent-team-run`, not within a forked skill context. The skill's job is to **generate the team config + stdin files**, then the Go binary handles orchestration.

### Revised Frontmatter

```yaml
---
name: review
description: >
  Orchestrated multi-domain code review. Generates team config and stdin files,
  then gogent-team-run spawns reviewers in parallel. Results via /team-result.
disable-model-invocation: true
argument-hint: "[--scope=glob | path | --all]"
allowed-tools: Read, Write, Glob, Grep, Bash(git diff *, git log *, git show *)
hooks:
  PostToolUse:
    - matcher: "Write"
      hooks:
        - type: command
          command: "$CLAUDE_PROJECT_DIR/.claude/hooks/team-launch-on-config.sh"
          async: true
          timeout: 600
---
```

### PostToolUse Hook: `team-launch-on-config.sh`

Shared hook (also used by updated `/braintrust`). Detects when a team config.json is written and launches `gogent-team-run`:

```bash
#!/bin/bash
# .claude/hooks/team-launch-on-config.sh
# PostToolUse hook — fires after Write tool
# Detects team config.json creation, launches gogent-team-run

INPUT=$(cat)
FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path')

# Only trigger for team config files in the sessions directory
if [[ "$FILE_PATH" != */teams/*/config.json ]]; then
  exit 0
fi

# Validate it's a real team config
WORKFLOW=$(jq -r '.workflow_type // empty' "$FILE_PATH" 2>/dev/null)
if [ -z "$WORKFLOW" ]; then
  exit 0
fi

TEAM_DIR=$(dirname "$FILE_PATH")
TEAM_NAME=$(basename "$TEAM_DIR")

# Launch gogent-team-run in background
gogent-team-run --config "$FILE_PATH" >> "$TEAM_DIR/runner.log" 2>&1 &
PID=$!

# Update config with background PID (atomic write)
TMPFILE=$(mktemp)
jq --arg pid "$PID" '.background_pid = ($pid|tonumber) | .status = "running" | .started_at = (now | strftime("%Y-%m-%dT%H:%M:%SZ"))' "$FILE_PATH" > "$TMPFILE" && mv "$TMPFILE" "$FILE_PATH"

echo "{\"systemMessage\":\"Team '$TEAM_NAME' launched (PID $PID). Use /team-status to monitor, /team-result to view output.\"}"
exit 0
```

### How This Fixes Reviewer JSON Propagation

Two layers of schemas exist but aren't connected to the runtime:

**Canonical validation schemas** (JSON Schema with `$schema`, `$id`, `required`, `additionalProperties: false`):
- `schemas/stdin/reviewer.json` — validates all reviewer stdin files
- `schemas/stdout/reviewer.json` — validates all reviewer stdout files
- `schemas/stdin/common-envelope.json` — common context fields (paths, capabilities, task_access)

**Domain-specific templates** (example documents with focus areas per reviewer type):
- `schemas/teams/stdin-stdout/review-backend.json` — security, API, concurrency focus areas
- `schemas/teams/stdin-stdout/review-frontend.json` — a11y, hooks, performance focus areas
- `schemas/teams/stdin-stdout/review-standards.json` — naming, complexity, DRY focus areas
- `schemas/teams/stdin-stdout/review-architect.json` — coupling, boundaries, testability focus areas

**The fix has two parts:**

**Part A: Skill generates stdin files**

The forked `/review` skill generates `stdin_{reviewer}.json` files for each reviewer:

1. Read `schemas/stdin/reviewer.json` for the required field structure
2. Read `schemas/teams/stdin-stdout/review-{domain}.json` for domain-specific `focus_areas`
3. Read `schemas/stdin/common-envelope.json` for context/capabilities injection
4. Populate dynamic fields:
   - `review_scope.files`: from git diff (filtered by domain relevance)
   - `git_context`: from git log, branch name, PR description
   - `context.project_root`: from `$CLAUDE_PROJECT_DIR`
   - `context.team_dir`: the team directory being created
   - `project_conventions`: from detected language + conventions file
5. Write to: `{team_dir}/stdin_{reviewer-name}.json`

**Part B: gogent-team-run validates against schemas**

When `gogent-team-run` processes agent output:
1. Extract structured JSON from CLI output (Layer 2 parsing)
2. Validate against `schemas/stdout/reviewer.json` before writing `stdout_{reviewer}.json`
3. If validation fails, log warning but still write (non-blocking — reviewer output is best-effort)
4. This requires `gogent-team-run` to embed or reference the JSON Schema for validation

**End result**: The canonical schemas at `schemas/stdin/` and `schemas/stdout/` become the runtime validation layer, not just documentation. The domain templates at `schemas/teams/stdin-stdout/` provide the focus area configuration per reviewer type.

## Part 5: Register `team-launch-on-config.sh` in Project Settings

The shared hook needs to be in `.claude/settings.json` (not skill frontmatter) so it fires regardless of which skill writes the config:

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "hooks": [
          {
            "type": "command",
            "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/team-launch-on-config.sh",
            "async": true,
            "timeout": 600
          }
        ]
      }
    ]
  }
}
```

**Why project settings, not skill frontmatter**: Multiple skills write team configs (`/braintrust`, `/review`, future `/implement`). A single hook in project settings catches all of them. Skill-scoped hooks only fire when that specific skill is active.

## Notes

### Why Not `context: fork` for `/braintrust`

Braintrust requires Mozart's interview phase, which needs user interaction in the main conversation. `context: fork` runs in isolation without conversation access. The interview must happen inline.

However, the EXECUTION phase (after interview, config generated) happens via the shared `team-launch-on-config.sh` hook — same mechanism as `/review`, just triggered from inline context instead of forked context.

### Stdin/Stdout Compatibility with Hook System

The hook system's stdin/stdout refers to **hook I/O** (JSON on stdin to hooks, JSON on stdout from hooks). The **reviewer stdin/stdout** refers to **agent I/O** (JSON files passed to/from reviewer agents via gogent-team-run).

These are separate channels:
- **Hook stdin**: `PostToolUse` event JSON → `team-launch-on-config.sh` reads it
- **Hook stdout**: Hook returns `systemMessage` JSON → Claude sees it
- **Reviewer stdin**: `stdin_backend-reviewer.json` file → `gogent-team-run` passes to agent CLI
- **Reviewer stdout**: Agent CLI JSON output → `gogent-team-run` extracts and writes `stdout_backend-reviewer.json`

No conflict. The hook launches the process that manages reviewer I/O.
