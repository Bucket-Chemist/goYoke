# TC-003: Fix Recursive Retry WaitGroup Panic

**Phase**: Phase 2: Go Binary Implementation
**Priority**: HIGH
**Blocked By**: TC-002 (mutex design needed for safe status updates)
**Effort**: Included in TC-008 (core binary implementation)

## Description

The original plan's `spawnAndWait()` function has `defer wg.Done()` at the top. On retry, it calls itself recursively, adding a second `defer wg.Done()`. Since `wg.Add(1)` was called only once, the WaitGroup counter goes negative, causing a panic on the second retry attempt.

This is a classic WaitGroup misuse pattern. The fix requires replacing recursion with iteration.

## Implementation Instructions

### Package Structure
- **Location**: `cmd/gogent-team-run/main.go` (part of TC-008 implementation)
- **Function**: `spawnAndWait()` - spawns claude CLI process and waits for completion

### Refactoring Pattern

**Current (broken) pattern:**
```
func spawnAndWait(...) {
    defer wg.Done()  // CALLED ONCE

    // ... spawn logic ...

    if err != nil && attempt < maxRetries {
        spawnAndWait(...)  // RECURSIVE - adds SECOND defer wg.Done()
    }
}
```

**Required pattern:**
```
func spawnAndWait(ctx context.Context, tr *TeamRunner, member *TeamMember, wg *sync.WaitGroup) {
    defer wg.Done()  // Single defer - matches single wg.Add(1) in caller

    for attempt := 0; attempt <= member.MaxRetries; attempt++ {
        // Update member status to "running" via mutex
        tr.updateMember(member.Name, func(m *TeamMember) {
            m.Status = "running"
            m.RetryCount = attempt
        })

        // Spawn Claude CLI process
        cmd, err := spawnClaude(ctx, tr, member)
        if err != nil {
            tr.updateMember(member.Name, func(m *TeamMember) {
                m.ErrorMessage = err.Error()
            })
            continue  // Retry loop continues
        }

        // Wait for process completion
        if err := cmd.Wait(); err != nil {
            tr.updateMember(member.Name, func(m *TeamMember) {
                m.ErrorMessage = err.Error()
            })
            continue  // Retry loop continues
        }

        // Validate stdout envelope
        if err := validateStdout(member); err != nil {
            tr.updateMember(member.Name, func(m *TeamMember) {
                m.ErrorMessage = err.Error()
            })
            continue  // Retry loop continues
        }

        // Success path - mark completed and return
        tr.updateMember(member.Name, func(m *TeamMember) {
            m.Status = "completed"
            m.CompletedAt = time.Now().Unix()
        })
        return
    }

    // All retries exhausted - mark failed
    tr.updateMember(member.Name, func(m *TeamMember) {
        m.Status = "failed"
    })
}
```

### Concurrency Safety

1. **WaitGroup semantics**: One `wg.Add(1)` in caller, one `defer wg.Done()` in callee
2. **Retry loop**: `for attempt := 0; attempt <= member.MaxRetries; attempt++` (0-indexed, inclusive upper bound)
3. **Status updates**: Always via `tr.updateMember()` which locks the mutex
4. **Early return**: On success, return immediately (no need to complete retry loop)

### Error Handling Pattern

- **Continue on error**: Each retry stage failure calls `continue` to advance retry loop
- **Final failure**: If loop completes without success, mark status as "failed"
- **Error context wrapping**: Use `fmt.Errorf("spawn attempt %d: %w", attempt, err)` for debugging

## Code Guidelines

From go.md conventions:

1. **Error wrapping**: `fmt.Errorf("spawn attempt %d: %w", attempt, err)` preserves error chain
2. **No recursion**: Iterative retry avoids stack depth and WaitGroup mismatches
3. **Mutex discipline**: All config/member mutations via `tr.updateMember()` helper
4. **Defer placement**: Single `defer wg.Done()` at top of function
5. **Context cancellation**: Check `ctx.Done()` before each retry attempt

### Edge Cases
