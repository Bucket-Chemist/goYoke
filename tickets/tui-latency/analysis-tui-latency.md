## Executive Summary

The GOfortress TUI pays 1.5-3.5 seconds of process initialization overhead per message because `query()` spawns a new CLI process each turn. Both analysts agree this is real, measurable, and worth fixing. However, the obvious solution -- migrating to the V2 session API (`unstable_v2_createSession`) -- is **blocked** by critical missing functionality: `SDKSession` exposes only 4 methods (`send`, `stream`, `close`, `[Symbol.asyncDispose]`) while the TUI actively depends on `interrupt()`, `setModel()`, `streamInput()`, `canUseTool()`, and ~15 Options fields that have no V2 equivalent. The highest-value immediate action is **V1.5: the AsyncIterable prompt pattern**, which keeps a single process alive across messages without touching V2 at all. This approach reuses the exact mechanism already proven in `useClaudeQuery.ts` lines 356-376 for built-in tool responses. Pin the SDK to `"0.2.31"` (exact) today, validate AsyncIterable empirically this week, and defer V2 migration until Anthropic ships the missing control surface.

## Problem Statement

### Original Request

The TUI has ~2x message latency (6-10s) compared to Claude Code CLI (3-5s). Investigate root cause and propose optimization path.

### Clarified Problem

Each call to `query()` in `useClaudeQuery.ts` (line 666) spawns a fresh `claude` CLI process. The `resume` option (line 671) restores conversation history but does not reuse the process. The 1.5-3.5s overhead is CLI startup: Node.js bootstrap, hook loading (`gogent-load-context`, `gogent-validate`), MCP server registration, and Anthropic API handshake. This overhead is paid N times for an N-message conversation.

### Success Criteria

1. First-message latency unchanged (process must start at least once)
2. Subsequent-message latency reduced to <1s overhead above API round-trip
3. All existing TUI features preserved: `interrupt()`, `setModel()`, `streamInput()` for built-in tools, `canUseTool()` permission modals, plan mode toggle, compaction detection
4. No regression in MCP server (`gofortress-interactive`) availability
5. Clean degradation path if optimization fails

## Analysis Perspectives

### Einstein (Theoretical Analysis)

**Focus**: Lifecycle impedance mismatch between stateless `query()` and stateful conversation; ownership model for persistent sessions; state machine design; alternative transport architectures.

**Key Insights**:

1. **Lifecycle impedance mismatch** is the root cause: `query()` treats each turn as a fresh invocation when conversations are inherently stateful. The TUI's architecture (React hooks + Zustand store) already models stateful sessions -- the SDK connector is the mismatch point.
2. **V1.5 AsyncIterable pattern** is the highest-value quick win. `query()` accepts `prompt: string | AsyncIterable<SDKUserMessage>`. Passing an AsyncIterable that yields messages on demand keeps a single CLI process alive indefinitely. The TUI already uses this exact mechanism for `streamInput()` tool responses (lines 357-376 of `useClaudeQuery.ts`).
3. **Module-level singleton** ownership (matching existing `mcpServer` pattern at `src/mcp/server.ts`) is the correct abstraction. React hooks consume session state; Zustand publishes it; neither owns the process.
4. **6-state machine**: UNINITIALIZED, CONNECTING, READY, STREAMING, ERROR, DEAD -- with message queueing during non-READY states. Generalizes TC-015a's single `pendingMessage` into a FIFO queue.
5. **V2 MCP gap**: `SDKSessionOptions` is missing `mcpServers` and `settingSources`. The `gofortress-interactive` MCP server is essential. If V2 cannot register MCP servers, migration is blocked regardless of other issues.
6. **Hybrid transport** (long-term): V2 primary with V1 fallback via circuit breaker pattern. Fast path (~50ms reconnect) with graceful degradation to slow path (~3-5s full startup).

**Root Cause Identified**: Per-message process spawning via `query()` with no process reuse mechanism.

**Novel Approaches Proposed**:

- **V1.5 AsyncIterable**: Keep single process alive by feeding messages through an async generator
- **Hybrid V2/V1 transport**: Circuit-breaker pattern with automatic fallback
- **Session singleton module**: Process ownership decoupled from React lifecycle

- A-1: V2 sessions support MCP registration -- **LOW confidence** (not in types)
- A-2: AsyncIterable keeps process alive between yields -- **MEDIUM confidence** (types support it, needs empirical test)
- A-3: CLI handles long-lived sessions without memory leaks -- **MEDIUM confidence** (no evidence either way)

### Resolved Divergence 2: Singleton Module vs Hook Refactor

**Einstein**: Proposes a module-level singleton (`SessionManager`) that owns the persistent process, analogous to the existing `mcpServer` pattern. React hooks consume state via Zustand subscriptions.

**Staff-Architect**: Warns that the 850-line `useClaudeQuery.ts` is already complex and adding a state machine inside it would be risky.

**Resolution**: Both are correct, and they point to the same architectural move. The `SessionManager` singleton IS the complexity extraction. The state machine lives in the singleton module (pure TypeScript, independently testable), not in the React hook. The hook becomes a thin adapter that subscribes to Zustand state published by the singleton.

This refactor addresses Staff-Architect's complexity concern while implementing Einstein's ownership model. The key insight: **moving process lifecycle out of the React hook and into a standalone module reduces hook complexity while enabling persistent sessions.**

File structure:

- `src/session/SessionManager.ts` -- Singleton, owns process, state machine, message queue
- `src/hooks/useClaudeQuery.ts` -- Thin adapter, subscribes to Zustand, delegates to SessionManager
- `src/store/slices/session.ts` -- Unchanged (already exists)

**Confidence**: HIGH

### Resolved Divergence 3: Testing Strategy

## Unified Recommendations

### Primary Recommendation: V1.5 AsyncIterable with SessionManager Extraction

Implement process reuse via the AsyncIterable prompt pattern, housed in a new `SessionManager` singleton module extracted from the existing `useClaudeQuery.ts` hook.

**Rationale**: This is the only approach that (a) addresses the latency problem, (b) uses proven SDK mechanisms, (c) preserves all existing TUI features, and (d) does not depend on unstable V2 API.

**Theoretical Support**: Einstein's lifecycle analysis proves the impedance mismatch. The AsyncIterable pattern resolves it at the correct abstraction level -- the SDK already supports this input type. The state machine provides formal guarantees about message ordering and error recovery.

**Practical Validation**: Staff-Architect's V2 gap inventory confirms that direct V2 migration is blocked. The existing `streamInput()` code (lines 356-376) proves AsyncIterable works for in-flight tool responses. The remaining question -- whether it works across turns -- is empirically testable in under a day.

### Secondary Recommendations

1. **Pin SDK version immediately**: Change `"^0.2.31"` to `"0.2.31"` in `package.json`.
   - Priority: HIGH (do today, zero risk)
   - Supports: Risk mitigation (C-3)

   - Priority: MEDIUM (do this week)
   - Supports: Long-term hybrid architecture

### Not Recommended

Revisit when SDK ships these capabilities.

2. **Modifying the Claude CLI**: Out of scope. The CLI is Anthropic's product. Work within SDK boundaries.

3. **WebSocket/IPC bypass of CLI**: Einstein considered and correctly dismissed this. The CLI provides hook execution, MCP registration, and settings loading that cannot be replicated externally without maintaining a parallel implementation.

## Open Questions

1. **Does the AsyncIterable generator keep the CLI process alive between yields?** The SDK types accept `AsyncIterable<SDKUserMessage>` as `prompt`, and the existing `streamInput()` code proves generators work for in-flight responses. But no one has tested whether the CLI process stays alive when the generator is idle between conversation turns. This is the single most important question and is answered by Phase 0 validation.

2. **What is the CLI's idle timeout behavior?** If the CLI has an internal idle timer (e.g., no activity for 60s triggers shutdown), persistent sessions may require a keepalive mechanism. Phase 0 validation should include a 60-second idle test.

3. **Can `resume` restore a session after process death in V1.5 mode?** The current code passes `resume: existingSessionId` (line 671). If the persistent process dies and a fresh `query()` is called with `resume`, does the new process correctly restore the full conversation? This determines the quality of error recovery.

4. **How does `canUseTool` interact with persistent sessions?** The callback is passed as an option to `query()` (line 678). In V1.5, `query()` is called once. Does the callback remain active for the entire session lifetime, or is it scoped to a single turn? If scoped, the permission modal flow breaks for subsequent messages.

5. **What is the memory profile of a 50+ message CLI session?** Each message adds to the conversation history. Over a long session, context window compaction may trigger frequently. Is there measurable RSS growth beyond the expected conversation buffer?
