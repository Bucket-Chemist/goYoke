## Executive Summary

Both problems -- TUI agent invisibility and premature timeout kills -- stem from a single architectural gap: the daemon treats agents as opaque processes with no feedback channel. The theoretically optimal solution replaces the fixed timeout with an activity-based deadline using `/proc/PID/io` watermarking (zero-cooperation monitoring) and extends `config.json`'s `Member` struct with health fields that the TUI already polls. The critical assumption -- that Claude CLI produces measurable `/proc/PID/io` changes during execution -- **MUST be empirically validated** before committing to the `/proc`-based approach.

**Recommended approach**: Phased implementation. Phase 1 adds per-member health fields to `config.json` and extends TeamDetail rendering (quick win, low risk). Phase 2 implements `/proc`-based activity monitoring contingent on empirical validation.

## Problem Statement

### Original Request

Two interconnected architectural gaps in the team-run daemon:
1. TUI agent panels show "No agents yet" despite active team-run agents, because the agent store is only populated from SDK events.
2. Hardcoded 5-minute timeout kills working agents via SIGKILL.

### Clarified Problem

The root cause is singular: **the daemon models agents as opaque processes with only two terminal states (completed/failed) and no intermediate observability.** The TUI visibility gap and the blind timeout are both symptoms of this missing feedback channel.

The solution requires:
- (a) A health monitoring mechanism that distinguishes "thinking" from "stalled" without agent cooperation
- (b) A data bridge from Go daemon health state to TypeScript TUI via the existing file-based channel
- (c) Replacement of the fixed timeout with an activity-based deadline

### Success Criteria

1. Daemon monitors per-agent health via observable side effects, not wall-clock time
2. TUI shows individual agent status (healthy/warning/stalled) from team-run in real time via `config.json` polling
3. Stalled agents are detected via inactivity (no I/O for threshold period), not fixed timeout
4. Graduated response: SIGTERM before SIGKILL, with configurable grace period
5. Working agents are never killed prematurely regardless of elapsed time
6. Solution extends existing patterns (`heartbeat.go` goroutine, `config.json` atomic writes, `useTeams.ts` polling)
7. Critical assumption validated: `/proc/PID/io` shows intermediate I/O changes during Claude CLI pipe execution

## Root Cause Analysis

### Cause 1: Opaque Process Model (Primary)

**Confidence**: high
**Affected**: `cmd/gogent-team-run/spawn.go`, `cmd/gogent-team-run/daemon.go`

The daemon models agents as opaque processes with only two terminal states (completed/failed). There is no intermediate observability -- no "alive and producing output", no "alive but stalled", no "alive and thinking".

The `select{}` in `executeSpawn` (spawn.go:221-251) has exactly three branches: context cancelled, process exited, timeout. There is no fourth branch for "check health".

```go
select {
case <-ctx.Done():        // Context cancelled
case err := <-waitDone:   // Process completed
case <-time.After(cfg.timeout):  // Fixed timeout -- KILLS process
}
```

This is **structurally incapable** of health-aware behavior. `time.After(cfg.timeout)` is a one-shot timer that fires once and kills the process. There is no periodic check, no reset mechanism, no activity-based extension.

### Cause 2: Disjoint Visibility Systems

**Confidence**: high
**Affected**: `packages/tui/src/hooks/useTeams.ts`, `packages/tui/src/components/AgentTree.tsx`

The TUI's agent visibility system (`AgentTree`, `store.agents`) and team visibility system (`TeamList`, `store.teams`) are completely disjoint data flows:
- **Agents** come from SDK `query()` events (interactive sessions only)
- **Teams** come from `config.json` polling (team-run daemon)

There is no bridge that maps a `config.json` Member to an Agent in the store. The Agent interface requires fields (`id`, `parentId`, `model`, `tier`, `status`) that exist in the Member data but are never extracted.

`AgentTree.tsx` line 127 shows "No agents yet" because the agents store is empty. `useTeams.ts` reads member status/cost but doesn't surface individual member data.

### Cause 3: Time-Health Conflation

**Confidence**: high
**Affected**: `cmd/gogent-team-run/spawn.go`

The timeout mechanism conflates "elapsed wall time" with "agent health". An LLM agent that is actively reasoning (consuming API tokens, building context) produces no observable stdout for 60-90 seconds at a time. A 5-minute timeout treats this legitimate thinking time as a stall.

**The fundamental category error is: time elapsed != lack of progress.**

TC-013c exposed this: implementation workers timed out at 5 minutes despite actively working. Increasing to 10 minutes was rejected because it doesn't solve the problem -- it merely delays the incorrect kill.

## Conceptual Framework: Observable Side Effects as Health Signals

Any process performing useful work must produce observable side effects. For an LLM agent, these side effects form a hierarchy of decreasing reliability:

| Signal | Source | Reliability | Notes |
|--------|--------|-------------|-------|
| **Network I/O** | `/proc/PID/io` read_bytes/write_bytes | High | CLI makes API calls = bytes read/written at socket level |
| **File I/O** | `/proc/PID/io` write_bytes, inotify | Medium | Agent creates/modifies files via Write/Edit tools |
| **Stdout bytes** | stdout buffer | Low | CLI produces batch JSON on exit, NOT streaming output |
| **CPU time** | `/proc/PID/stat` utime+stime | Low | CLI is I/O-bound; near-zero CPU during API waits is healthy |

### Key Insights

1. **Zero-cooperation monitoring**: The daemon doesn't need the agent's cooperation. `/proc` provides a monitoring channel that the daemon can read on a ticker without any changes to the Claude CLI.

2. **`/proc/PID/io` watermarking**: `read_bytes` and `write_bytes` are monotonically increasing counters. If they change between monitoring intervals, the process is performing I/O -- which for an LLM agent means API communication.

3. **Thinking vs stalled**: The distinction is actually between "network-active but stdout-quiet" (thinking/reasoning) and "network-inactive and stdout-quiet" (truly stalled). An LLM agent that is reasoning is still making API calls.

4. **Stdout is batch, not streaming**: Claude CLI with `-p --output-format json` produces a JSON array as its final output, not a stream. Stdout activity is a terminal signal, not a periodic health signal.

5. **config.json as data plane**: Already serves as a one-way daemon-to-TUI data channel with atomic writes and adaptive polling. Adding health fields to the Member struct requires no new files, no new protocols, and no changes to polling infrastructure.

## First Principles Analysis

### Assumptions Challenged

| Assumption | Validity | Evidence | Implication if Wrong |
|-----------|----------|----------|---------------------|
| Fixed timeout is necessary for lifecycle management | **Invalid** | Task complexity varies more than model speed. The correct abstraction is an activity deadline, not an elapsed-time deadline. | A hard ceiling should remain for cost control (30m+ for Opus), but shouldn't be the primary kill mechanism. |
| Stdout activity is a reliable health signal | **Questionable** | Claude CLI `-p --output-format json` produces batch output on exit, not streaming. Stderr may have undocumented output. | Must rely on lower-level signals (`/proc/PID/io`) rather than stdout bytes. |
| File system is the only viable daemon-to-TUI channel | **Valid** | Go daemon and TypeScript TUI are separate processes with no shared memory or sockets. File-based communication is observable, debuggable, and crash-recoverable. | If sockets existed, push-based updates would be superior, but the infrastructure cost is disproportionate. |
| Process CPU time indicates useful work | **Questionable** | Claude CLI is I/O-bound. Near-zero CPU during API waits is healthy behavior, not a stall. | Using CPU time as a health signal would falsely flag API-waiting agents as stalled. |

### Fundamental Constraints

1. Claude CLI produces batch output (JSON array on exit), not streaming. Health signals must come from the OS process layer (`/proc`) or file system layer.
2. The daemon and TUI communicate via files. Health state must be written to `config.json` by the daemon and read by the TUI on its polling cycle.
3. Multiple agents run concurrently within a wave. Health monitoring must be per-agent, not per-wave.
4. SIGKILL is unrecoverable. The daemon should exhaust softer interventions (SIGTERM, extended waiting) before SIGKILL.
5. The monitoring interval creates a resolution floor. 10s balances detection speed with write frequency.
6. `/proc/PID/io` requires same UID. Since the daemon spawns the CLI processes, they share a UID -- works without privilege escalation.

## Novel Approaches Proposed

### 1. Adaptive Activity Deadline with `/proc/PID/io` Watermarking

**Feasibility**: high

Replace the monolithic `select{}` in `executeSpawn` with a health-monitored loop:
1. Start a `healthMonitor` goroutine that reads `/proc/PID/io` every 10s and updates a shared `lastActivityAt` atomic value
2. Replace `time.After(timeout)` with a ticker that checks `lastActivityAt` against an inactivity threshold
3. The select now has four cases: `ctx.Done`, `waitDone`, `ticker.C` (check health), absolute-ceiling (hard max, e.g. 30 minutes)
4. The health goroutine stops when the process exits

**Pros**: Zero-cooperation, high fidelity, adaptive, Linux-native, composable with heartbeat.go pattern
**Cons**: Linux-only (acceptable -- project targets Linux exclusively), 10s granularity floor, false negatives possible for agents stuck in failing API retry loops
**Risks**: `/proc/PID/io` shows process I/O only (not children), zombie processes may make it unreadable (mitigated by existing `waitDone` channel)

### 2. Config.json Member Health Fields as Unified Data Model

**Feasibility**: high

Add to Member struct:
```go
type Member struct {
    // ... existing fields ...
    HealthStatus   string  `json:"health_status"`    // healthy|warning|stalled|unknown
    LastActivityAt *string `json:"last_activity_at"` // ISO8601 timestamp
    LastCheckedAt  *string `json:"last_checked_at"`  // ISO8601 timestamp
    IoReadBytes    *int64  `json:"io_read_bytes"`    // from /proc/PID/io
    IoWriteBytes   *int64  `json:"io_write_bytes"`   // from /proc/PID/io
}
```

The daemon's health goroutine calls `updateMember()` with these fields every monitoring interval. The TUI's `useTeams.ts` extracts these fields and surfaces them in TeamDetail.

**Pros**: No new files or protocols, extends proven pattern, additive schema change (backward compatible)
**Cons**: Increases config write frequency, config.json becomes both control plane and data plane (mild SRP violation)

### 3. Two-Tier Stall Detection: Activity Watermark + Escalating Response

**Feasibility**: high

Three-state health model:
- **healthy**: I/O delta > 0 in last check
- **warning**: No I/O delta for N checks (model-adaptive: Haiku 30s, Sonnet 60s, Opus 120s)
- **stalled**: No I/O delta for M checks (model-adaptive: Haiku 60s, Sonnet 120s, Opus 300s)

On transition to "stalled": SIGTERM. After grace period: SIGKILL. All transitions logged in `config.json` for TUI display and post-mortem analysis.

**Pros**: Graduated response, warning state provides TUI visibility, mirrors existing `killAllChildren()` two-phase pattern
**Cons**: More state machine complexity, threshold tuning is empirical, warning state may cause user anxiety

### 4. TUI Agent Bridge via Member-to-Agent Projection

**Feasibility**: medium

In `useTeams.ts`, after parsing team summaries, iterate over running teams' members. For each member with `status` running/completed/failed, upsert an Agent into the agents store with synthetic IDs:
```typescript
{
    id: `team-${dir}-${member.name}`,
    parentId: `team-${dir}-root`,
    model: member.model,
    status: mapStatus(member.status),
    pid: member.process_pid,
    cost: member.cost_usd
}
```

**Pros**: Reuses AgentTree rendering, clean separation (useTeams handles data, AgentTree handles rendering)
**Cons**: Mixes two Agent data sources (SDK + file polling) in one store, requires synthetic IDs, status mapping imperfect

### 2. Auto-kill vs user-initiated kill for stalled agents

**Status**: Resolved
**Confidence**: high

**Resolution**: Default to conservative behavior: the health monitor **REPORTS** status (healthy/warning/stalled) to `config.json` but does **NOT** auto-kill. The existing hard ceiling timeout provides the safety net. Users can manually kill via `/team-cancel`. Auto-kill can be added as an opt-in configuration once health signal reliability is proven.

**Reasoning**: The cost of a false positive (killing a working Opus agent mid-analysis, losing $1+ of work) exceeds the cost of a false negative (a truly stalled agent running an extra few minutes).

### Primary Recommendation

**Implement unified agent lifecycle monitoring in two phases: Phase 1 (health fields + TUI visibility) delivers immediate value with low risk. Phase 2 (`/proc`-based activity monitoring) replaces the fixed timeout contingent on empirical validation.**

**Rationale**: Phased delivery isolates the unvalidated `/proc` assumption to Phase 2 while delivering TUI visibility and basic health reporting in Phase 1. This respects the missing Staff-Architect review by not committing to the highest-risk component until it's validated.
