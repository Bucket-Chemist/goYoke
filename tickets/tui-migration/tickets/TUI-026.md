# TUI-026: Status line data wiring

## Description

Wire real data into the StatusLineModel from internal/tui/components/statusline/statusline.go. The status line should display 8 fields: (1) SessionCost from CostTracker.GetSessionCost(), (2) TokenCount from session state, (3) ContextPercent calculated from token count / context window, (4) PermissionMode from session state, (5) ActiveModel from session state (updated on system.init and model switch), (6) Provider from provider state (active provider name), (7) GitBranch from periodic exec.Command('git rev-parse --abbrev-ref HEAD') via tea.Every(30s), (8) AuthStatus from periodic exec.Command('claude auth status --json') via tea.Every(60s). Create two tick commands: GitBranchTickMsg and AuthStatusTickMsg that run the respective commands in background goroutines and send results as tea.Msg. Never block Update() or View() with subprocess calls. Handle command failures gracefully (show 'N/A' or empty). Update the status line on: ResultMsg (cost, tokens), SystemInitMsg (model, permission mode), StatusUpdateMsg (permission mode), provider switch, git branch tick, auth status tick.

## Review Amendments

**⚠️ Review m-6:** `git rev-parse` and `claude auth status` run via subprocess every 30s/60s respectively. On systems without git or claude CLI in PATH, these will fail silently — add graceful fallback: check PATH before spawning, cache results, show 'N/A' if binary not found. This is acceptable for a dev tool but prevents unnecessary subprocess spawn failures.
