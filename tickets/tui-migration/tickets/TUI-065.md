# TUI-065: Skeleton loading screens

## Description

Add skeleton loading screens that display during slow component initialization. The skeleton provides visual structure before content is ready, reducing perceived loading time.

Create `internal/tui/components/skeleton/` package:
```go
package skeleton

type SkeletonVariant int
const (
    SkeletonConversation SkeletonVariant = iota
    SkeletonAgentTree
    SkeletonSettings
    SkeletonDashboard
)

type SkeletonModel struct {
    variant SkeletonVariant
    width   int
    height  int
    frame   int // current animation frame for shimmer
}

// View renders the skeleton for the given variant.
// The shimmer effect moves a lighter block left-to-right across the skeleton.
func (m SkeletonModel) View() string
```

Four skeleton variants with "shimmer" animation:
1. **Conversation**: horizontal lines of varying width (mimics chat messages)
2. **AgentTree**: indented lines (mimics tree nodes)
3. **Settings**: key-value pairs (mimics settings rows)
4. **Dashboard**: metric cards (mimics dashboard blocks)

**Shimmer:** A lighter-colored block slides across each skeleton line using the `AnimateTickMsg` from TUI-064. The shimmer moves left-to-right and loops.

**When to show:** Skeleton screens are shown only when CLI initialization exceeds 500ms. Components show their skeleton variant during the loading window, then switch to their real content.

**Note from review (M-4):** The TUI loads in < 100ms under normal conditions. Skeleton screens are shown only when CLI initialization exceeds 500ms threshold, not on every startup.

## Review Amendments

**M-4 (Major — should address):** The review noted that skeleton screens solve a problem that does not exist for sub-100ms startups. The implementation must include the 500ms threshold guard: skeletons are shown only when `CLIInitDuration > 500ms`. This makes them visible only during slow network/disk conditions while having zero overhead for normal startup.
