# TUI-004: Spike - Side channel IPC POC (MCP server to TUI)

## Description

Build a proof-of-concept demonstrating Unix domain socket IPC between a Go MCP server process and a Go TUI process. This validates the side channel design for interactive tool results. Steps: (1) Create a minimal TUI program (cmd/gofortress-ipc-poc/tui/main.go) that listens on a Unix domain socket at $XDG_RUNTIME_DIR/gofortress-poc-{pid}.sock. (2) Create a minimal MCP-like server (cmd/gofortress-ipc-poc/mcp/main.go) that connects to the socket path passed via GOFORTRESS_SOCKET env var. (3) Define a simple JSON protocol: MCP sends {type: 'modal_request', id: 'req-1', message: 'Allow Write?', options: ['Allow', 'Deny']}. TUI responds with {type: 'modal_response', id: 'req-1', value: 'Allow'}. (4) In the TUI, receive the request via the socket, inject it as a tea.Msg via Program.Send(), handle it in Update(), and send the response back over the socket. (5) Verify the round-trip works and measure latency (<1ms expected for UDS). (6) Handle connection lifecycle: MCP connects after TUI is ready, reconnect on failure. (7) Document the protocol and patterns in tickets/tui-migration/spike-results/ipc-protocol.md.

## Acceptance Criteria

- [ ] UDS communication works between two Go processes
- [ ] Modal request/response round-trip completes in <5ms
- [ ] tea.Msg injection via Program.Send() from socket reader goroutine works
- [ ] Socket cleanup on TUI exit (no stale sockets)
- [ ] Connection retry logic in MCP server side works
- [ ] Protocol document written to tickets/tui-migration/spike-results/ipc-protocol.md
