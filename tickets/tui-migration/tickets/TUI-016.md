# TUI-016: Wire CLI driver + MCP server + IPC bridge into startup sequence

## Description

Wire all Phase 3 components into the root AppModel startup sequence. Modify internal/tui/model/app.go Init() and Update() to implement the startup sequence: (1) IPCBridge.Start() -- creates UDS listener (immediate), (2) Spawn Claude CLI with --input-format stream-json --output-format stream-json --include-partial-messages --permission-mode acceptEdits --allowedTools "mcp__gofortress__*" --mcp-config pointing to gofortress-mcp binary, GOFORTRESS_SOCKET env var set to bridge socket path. Note: --allowedTools is required because acceptEdits does NOT auto-approve MCP tools (TUI-002 spike finding). Claude CLI will spawn the MCP server as a child process. (3) CLIDriver.Start() -- begins consuming NDJSON events. (4) Wait for SystemInitMsg from CLI (confirms CLI is ready, MCP tools discovered). (5) Set AppModel.ready = true, show the TUI. Add new message types: CLIReadyMsg (after SystemInitMsg processed), StartupErrorMsg (if any component fails). Handle CLIDisconnectedMsg with error classification and reconnection logic (max 3 attempts with exponential backoff). Handle the re-subscription pattern: after each CLI event is processed in Update(), return CLIDriver.WaitForEvent() as a Cmd to keep listening. Permission handling uses the hybrid approach from TUI-001 spike (Option D): acceptEdits auto-approves Write/Edit/Bash (show diffs post-hoc), while interactive tools (AskUserQuestion, EnterPlanMode) are handled via MCP side-channel — the MCP server receives these as tool invocations and forwards to the TUI via UDS modal requests. See spike-results/permission-protocol.md §5 for full architecture.
