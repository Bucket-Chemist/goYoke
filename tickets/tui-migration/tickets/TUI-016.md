# TUI-016: Wire CLI driver + MCP server + IPC bridge into startup sequence

## Description

Wire all Phase 3 components into the root AppModel startup sequence. Modify internal/tui/model/app.go Init() and Update() to implement the startup sequence: (1) IPCBridge.Start() -- creates UDS listener (immediate), (2) Spawn Claude CLI with --mcp-config pointing to gofortress-mcp binary, --permission-prompt-tool stdio, GOFORTRESS_SOCKET env var set to bridge socket path. Claude CLI will spawn the MCP server as a child process. (3) CLIDriver.Start() -- begins consuming NDJSON events. (4) Wait for SystemInitMsg from CLI (confirms CLI is ready, MCP tools discovered). (5) Set AppModel.ready = true, show the TUI. Add new message types: CLIReadyMsg (after SystemInitMsg processed), StartupErrorMsg (if any component fails). Handle CLIDisconnectedMsg with error classification and reconnection logic (max 3 attempts with exponential backoff). Handle the re-subscription pattern: after each CLI event is processed in Update(), return CLIDriver.WaitForEvent() as a Cmd to keep listening. Handle permission control_request messages: if the NDJSON parser encounters a control_request type (from --permission-prompt-tool stdio), create a PermissionRequestMsg and handle it like a modal request. Write the response to CLI stdin. This is Option A from the strategy -- if spike shows it's not feasible, fall back to MCP-mediated permissions (Option B) where the MCP server handles permissions via UDS modal requests.

## Acceptance Criteria
