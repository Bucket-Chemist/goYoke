# TUI-034: Graceful shutdown with timing budget

## Description

Implement graceful shutdown in internal/tui/lifecycle/shutdown.go and wire into root AppModel. ShutdownManager struct: program *tea.Program, cliDriver *CLIDriver, bridge *IPCBridge, sessionSaver func(). Shutdown sequence with timing budget (total 10s max): (1) Save session state immediately (sessionSaver(), <100ms). (2) Interrupt active CLI query -- send SIGINT, wait up to 2s (cliDriver.Interrupt(), timer.After 2s). (3) If CLI still running after 2s, send SIGKILL. (4) Shutdown IPC bridge (bridge.Shutdown(), closes socket, resolves pending modals with cancel). (5) Wait for Go hooks completion (500ms -- hooks like gogent-archive run on session end). (6) Background team agents: DO NOT wait. They are detached processes managed by gogent-team-run. List them as 'still running' in final status but don't block. (7) tea.Quit to exit the program. Handle SIGINT (Ctrl+C): first press triggers graceful shutdown. Second press within 2s forces immediate exit (tea.Quit without cleanup). Handle SIGTERM: immediate graceful shutdown. Wire into root AppModel: intercept ForceQuit (Ctrl+C) keybinding. First Ctrl+C sets shutdownInProgress=true and starts shutdown. Second Ctrl+C forces exit. Show 'Shutting down...' in status line during shutdown.

## Post-Phase 7 Review: DES-1 Requirements (2026-03-23)

The post-Phase 7 code review (Staff Architect: APPROVE_WITH_CONDITIONS) identified shutdown orchestration gaps that must be addressed in this ticket. These are prerequisites for Phase 8–9 readiness.

### Required: Wire ProcessManager.StartSignalHandler into main.go

`internal/lifecycle/process.go` already provides `ProcessManager` with signal handling. Wire `StartSignalHandler()` into `cmd/gofortress/main.go` so SIGINT/SIGTERM are caught at the process level, not just the Bubble Tea key level. The current defer-based approach in main.go does not guarantee cleanup ordering.

### Required: Replace defer-based shutdown with explicit sequenced shutdown

The current shutdown path uses `defer` chains which execute in LIFO order and do not enforce the timing budget defined in this ticket's Description. Replace with an explicit sequenced shutdown function that:

1. Saves session state (`sessionSaver()`, deadline 100ms)
2. Interrupts active CLI query (SIGINT → 2s wait → SIGKILL)
3. Shuts down IPC bridge (`bridge.Shutdown()`)
4. Waits for Go hooks (500ms budget)
5. Calls `tea.Quit`

The LIFO ordering of the current defer chain is wrong: bridge.Shutdown must come after CLI interrupt (not before), because pending modal resolutions may still arrive from the CLI during its 2s wind-down.

### Required: SIGKILL escalation goroutine must complete before process exit

If the 2s CLI wind-down timer fires and sends SIGKILL, the escalation goroutine must be joined (or its context cancelled) before the process exits. Currently there is no join point, which can leave the goroutine running after main() returns on some Go runtime versions.

### Required: Fix bridge/driver shutdown ordering

Current ordering (as inferred from code review): bridge closes first, then driver. Correct ordering: driver closes first (so no new IPC messages arrive), then bridge closes (processes any in-flight messages, then terminates). Reversing this prevents a race where the bridge receives a message after its socket has been torn down.
