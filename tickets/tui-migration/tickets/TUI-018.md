# TUI-018: Permission flow implementation

## Description

Implement the full permission flow in internal/tui/components/modals/permission.go. This handles all 5 permission variants from the migration plan Section 14. (1) Standard tool permission (can_use_tool): Present modal with options [Allow, Allow Always, Deny]. If Deny, present follow-up Input modal for deny reason. (2) EnterPlanMode: Present Confirm modal 'Enter plan mode?'. (3) ExitPlanMode: Present Ask modal with options [Approve Plan, Request Changes, Reject Plan]. If Request Changes, present Input modal for feedback. Load plan markdown from file and set PlanPreviewContent. (4) AskUserQuestion: Handle 1-4 sequential Ask modals. The MCP server or control protocol may send multiple questions. (5) acceptEdits auto-approve: If permission mode is acceptEdits, automatically approve without showing modal. Create a PermissionHandler struct that orchestrates these multi-step flows. The handler receives PermissionRequestMsg (from control_request parsing or MCP tool handler), determines the flow type, queues the appropriate modal(s), and sends the final decision back (either to CLI stdin for Option A, or via UDS response for Option B). Wire into the root AppModel: on PermissionRequestMsg, delegate to PermissionHandler which manages the modal queue for multi-step flows. Test with mock permission requests for each variant.
