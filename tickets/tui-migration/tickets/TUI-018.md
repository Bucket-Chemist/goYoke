# TUI-018: Permission flow implementation

## Description

Implement the full permission flow in internal/tui/components/modals/permission.go. TUI-001 spike confirmed: there is NO control_request protocol in stream-json mode. The architecture uses Option D (hybrid) from spike-results/permission-protocol.md §5. Two permission paths exist:

**Path 1 — acceptEdits (Write/Edit/Bash):** These tools are auto-approved by --permission-mode acceptEdits. The TUI displays tool activity via stream_event parsing and shows diffs post-hoc via the tool_use_result.structuredPatch field in user events. No modal needed.

**Path 2 — MCP side-channel (AskUserQuestion, EnterPlanMode, ExitPlanMode):** These interactive tools are registered as MCP tools on the Go MCP server (TUI-014). When Claude invokes them, the MCP server forwards the request to the TUI via UDS (TUI-015). The TUI shows the appropriate modal and returns the result via UDS back to the MCP server, which returns it to Claude.

Permission variants: (1) EnterPlanMode: Present Confirm modal 'Enter plan mode?'. Response returned via MCP tool result. (2) ExitPlanMode: Present Ask modal with options [Approve Plan, Request Changes, Reject Plan]. If Request Changes, present Input modal for feedback. Load plan markdown from file and set PlanPreviewContent. (3) AskUserQuestion: Handle 1-4 sequential Ask modals from the MCP tool input (questions array with options). (4) Standard tool display: For Write/Edit/Bash, show post-hoc diff display using tool_use_result data from NDJSON stream (not a modal, just UI update).

Create a PermissionHandler struct that orchestrates multi-step flows. The handler receives PermissionRequestMsg (from MCP tool handler via UDS bridge), determines the flow type, queues the appropriate modal(s), and sends the final decision back via UDS response. Wire into the root AppModel: on PermissionRequestMsg, delegate to PermissionHandler which manages the modal queue for multi-step flows. Test with mock permission requests for each variant.
