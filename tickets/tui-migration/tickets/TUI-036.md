# TUI-036: Component unit tests (teatest)

## Description

Create comprehensive unit tests for all Bubble Tea components using the charmbracelet/x/exp/teatest package (or direct function call testing). Files: internal/tui/components/*/..._test.go for each component. Test strategy: (1) For each component, test Update() with mock messages and verify state changes. (2) Test View() output contains expected strings. (3) Test keyboard handling (send tea.KeyMsg, verify state transition). (4) For the root AppModel, test the full startup sequence: WindowSizeMsg -> ready=true, CLI events -> state updates, key routing -> correct child receives input. Create test fixtures in internal/tui/testdata/: sample NDJSON events (system.init, assistant, result), sample agent tree data, sample team configs. Test the re-subscription pattern: after each CLI event, verify WaitForEvent cmd is returned. Test modal queue: send multiple ModalRequestMsg, verify sequential processing. Test auto-scroll behavior: add messages, verify viewport at bottom, scroll up, verify auto-scroll disabled, add message at bottom, verify auto-scroll re-enabled.

### Cross-Layer TestHarness Struct

Create `internal/tui/testutil/harness.go` with a `TestHarness` struct that wires a pipe-injected CLI source into the real `AppModel` and provides assertions on rendered view output. This enables integration-level testing without spawning a real Claude CLI process.

```go
type TestHarness struct {
    model    model.AppModel
    cliPipe  *io.PipeWriter  // inject synthetic NDJSON events
    program  *tea.Program
}

func NewTestHarness(t *testing.T) *TestHarness { ... }
func (h *TestHarness) SendCLIEvent(event string) { ... }
func (h *TestHarness) AssertViewContains(t *testing.T, substr string) { ... }
func (h *TestHarness) AssertState(t *testing.T, check func(model.AppModel) bool) { ... }
```

### Why This Is Now Feasible

The DES-2 refactor (post-Phase 7 review) extracted all widget interfaces into `internal/tui/model/interfaces.go`. Every component that `AppModel` depends on is now accessed via an interface, making all widgets mockable for harness construction.

### Minimum Integration Scenarios to Cover

1. Inject a `system.init` NDJSON event → assert status line shows "Ready"
2. Inject an `assistant` event with a text block → assert claude panel view contains the text
3. Inject a `tool_use` event → assert agent tree registers a new agent entry
4. Inject a `result` event → assert cost tracker reflects the reported cost
5. Send a modal request via the bridge → assert modal view is displayed → inject modal response → assert modal dismissed

These scenarios verify the full path: pipe → CLIDriver → AppModel.Update() → View() output, without mocking the model itself.
