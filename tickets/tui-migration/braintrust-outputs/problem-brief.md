## 3. Analysis Axes

### For Einstein (Theoretical)

- **Primary question**: Is the three-process topology (Go TUI + TS MCP sidecar + Claude Code CLI) the right architecture, or does a simpler topology achieve the same goals with less operational risk?
- **Conceptual focus**:
  - IPC design: HTTP bridge vs. Unix domain sockets vs. stdio pipes vs. gRPC
  - State model translation: Can 9 Zustand slices collapse cleanly into Elm architecture without semantic loss?
  - Process lifecycle coordination: What happens when one of three processes dies? Restart semantics, state recovery
  - The SessionManager AsyncGenerator pattern vs. Go's channel+goroutine subprocess driver -- are there edge cases lost in translation?
  - NDJSON protocol reliability: undocumented event types, schema evolution, backward compatibility
- **Novel angles**:
  - Could Go's MCP SDK (modelcontextprotocol/go-sdk, already in go.mod) replace the TS sidecar entirely?
  - Could the sidecar be embedded via wasm/wasi instead of a separate Node.js process?
  - Is the HTTP bridge necessary, or could the Go TUI communicate with the sidecar via stdin/stdout MCP protocol directly?
  - What if the TS sidecar is unnecessary and the Go TUI implements MCP tools natively?
- **First principles**:
  - Why does the MCP server need to be in TypeScript at all? The stated reason is "createSdkMcpServer tracks Claude Code CLI updates" -- how true is this in practice?
  - The plan assumes Bubble Tea v1 over v2 -- is this still the right call given v2 has been out since Feb 2026?
  - The plan keeps the process registry in the TS sidecar -- but if Go drives the TUI, shouldn't process management be in Go?

### For Staff-Architect (Practical)

- **Review focus**: Validate every assumption in the migration plan against actual code
- **Risk areas**:
  - The plan's NDJSON event type definitions vs. actual events emitted by Claude CLI
  - Modal queue timing: HTTP round-trip latency on permission flows that currently run in-process
  - Process registry split: spawn tracking in TS sidecar while TUI display is in Go -- race conditions
  - The plan says "TS MCP sidecar changes are minimal" -- verify this claim against actual tool implementations
  - Multi-provider support: the plan barely mentions it, but the existing code has per-provider session IDs, models, and message history
- **Constraint check**:
  - Can the TS sidecar really be bundled as a single esbuild file?
  - Does the Go MCP SDK in go.mod provide enough to replace createSdkMcpServer?
  - Are there Claude CLI flags or behaviors the plan assumes but that may not exist?
- **Ticket reconciliation**:
  - Map each of the 13 old tickets to the new 56-ticket plan
  - Identify overlap, conflicts, and gaps
  - Assess whether old ticket granularity or new ticket granularity is more appropriate
