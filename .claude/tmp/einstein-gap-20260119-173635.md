## Problem Statement

**Current ticket (GOgent-028)** specifies implementing handoff document generation as **markdown files** (`.claude/memory/last-handoff.md`), but this appears architecturally inconsistent with the project's stated goals of:
- Machine-readable data storage
- Best-in-class data engineering
- Structured, queryable memory systems

**Core Question**: Why use markdown for handoff documents when the rest of the memory system is moving toward structured, machine-readable formats (JSONL, potentially databases)?

## Architectural Tension

### Markdown Approach (Current Ticket)

**Pros:**
- Human-readable
- Easy to review in session handoff
- Git-friendly (diffs are readable)
- Low implementation complexity
- Claude can read it naturally via Read tool

**Cons:**
- Not machine-queryable
- No schema enforcement
- Difficult to aggregate across sessions
- Parsing required for programmatic access
- Not composable with data engineering tooling
- Inconsistent with JSONL storage for learnings/violations

### Structured Format Alternative (Not Specified)

**Pros:**
- Machine-queryable (SQL, JSON path queries)
- Schema-enforced (validation, type safety)
- Aggregatable across sessions
- Composable with analytics tools
- Consistent with broader memory architecture
- Enables time-series analysis, trend detection

**Cons:**
- Less human-readable (requires tooling)
- Higher implementation complexity
- May need dual output (structured + human view)
- Git diffs less readable

## Specific Questions for Einstein

### Primary Question

**What is the optimal storage format for session handoff documents in a system optimized for machine-readable, best-in-class data engineering?**

### Sub-Questions

1. **Format Decision**:
   - Should handoff be JSON/JSONL instead of markdown?
   - Should there be dual output (structured + human-readable view)?
   - Is markdown acceptable if it's the *only* human-facing component?

2. **Schema Design**:
   - If structured: what schema best captures session context?
   - How to represent "action items" and "context" in structured form?
   - Should schema evolve over time (versioning)?

3. **Tooling Trade-offs**:
   - Is the human-readability of markdown worth the loss of queryability?
   - Could we generate markdown *from* structured data as a view?
   - What's the right balance between developer experience and data engineering goals?

4. **Integration Points**:
   - How should handoff integrate with load-routing-context hook?
   - Should handoff be queryable by agents (e.g., "what did last session do?")?
   - How does this affect session replay/debugging workflows?

5. **Future-Proofing**:
   - If we start with markdown, what's the migration path to structured?
   - If we start with structured, can we add markdown views later?
   - Which decision minimizes technical debt?

6. **Consistency Principle**:
   - Should handoff format match the rest of memory system (JSONL)?
   - Or is handoff special because it's the human-facing summary?
