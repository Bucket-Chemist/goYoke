## Executive Summary

The TUI's core deficiency is an **architectural data flow gap**: the agents store exists and works correctly, but nothing populates it. The `useClaudeQuery` hook processes SDK events but never extracts agent information or calls `addAgent()`. This is not a rendering bug - it's a missing integration layer. The conversation scrolling issues stem from Viewport's item-based pagination conflicting with variable-height message rendering. The solution requires: (1) adding agent lifecycle tracking to the SDK event handler, (2) creating an activity slice for real-time tool call streaming, (3) fixing Viewport to use line-based scrolling, and (4) implementing proper master-detail data flow for agent selection.

## Analysis Perspectives

### Einstein (Theoretical Analysis)

**Focus**: Event-driven architecture for agent lifecycle, data flow patterns, conceptual frameworks

#### Root Cause Analysis

**Surface Problem**: AgentTree shows "No agents yet"

**Underlying Cause**: The `handleSystemEvent` and `handleAssistantEvent` functions in `useClaudeQuery.ts` process SDK events but extract only session metadata and message content - they don't extract agent information.

**Fundamental Issue**: **Missing event-to-state mapping layer**. The system has:
- Event source (SDK query stream)
- State destination (agents Zustand slice)
- But no connector between them

**Evidence Chain**:
1. `SystemEvent` type (events.ts:7-23) includes `agents?: Array<{id, model, tier, description}>`
2. `handleSystemEvent` (useClaudeQuery.ts:182-202) only extracts `session_id` and `model`
3. `addAgent` (agents.ts:17) is never called from useClaudeQuery
4. AgentTree correctly renders when agents exist (tested via manual store manipulation)

#### Conceptual Framework: Event Sourcing Pattern

The TUI should treat SDK events as an **event stream** that drives multiple state projections:

```
SDK Events
    │
    ├──► Messages Projection (current: working)
    │         └── addMessage(), updateLastMessage()
    │
    ├──► Session Projection (current: working)
    │         └── updateSession(), incrementCost()
    │
    ├──► Agents Projection (current: MISSING)
    │         └── addAgent(), updateAgent()
    │
    └──► Activity Projection (current: MISSING)
              └── addActivity(), updateActivity()
```

Each projection is independent but derives from the same event stream.

#### First Principles Analysis

**Axiom 1**: UI state should be a pure function of events received
**Axiom 2**: Events carry all information needed to reconstruct state
**Axiom 3**: Different UI panels may need different projections of the same events

**Derived Implications**:
- Agent data exists in SDK events (SystemEvent.agents field) - we just don't extract it
- Tool calls exist in AssistantEvent.content_blocks - we render them but don't track by agent
- The same event stream should feed multiple independent state slices

#### Novel Approaches

**Approach 1: Event Bus Pattern**
Create an internal event bus that SDK events flow through, with multiple subscribers:
```typescript
const eventBus = {
  emit: (event: SDKEvent) => {
    messagesHandler(event);
    agentsHandler(event);
    activityHandler(event);
  }
};
```
*Rationale*: Clean separation, easy to add new projections
*Tradeoff*: Additional abstraction layer, slight complexity increase

**Approach 2: Unified Event Handler**
Extend existing handleSystemEvent/handleAssistantEvent to populate all slices:
```typescript
const handleSystemEvent = (event: SDKSystemEvent) => {
  // Existing: session update
  updateSession({ id: event.session_id });

  // NEW: agent population
  if (event.agents) {
    event.agents.forEach(agent => addAgent({...}));
  }
};
```
*Rationale*: Minimal change, no new abstractions
*Tradeoff*: Handler becomes large, harder to test independently

**Approach 3: Derived State Pattern**
Don't store agents separately - derive from message stream:
```typescript
const agents = useMemo(() =>
  extractAgentsFromMessages(messages),
  [messages]
);
```
*Rationale*: Single source of truth (messages)
*Tradeoff*: Performance concern with large message histories, harder to track agent lifecycle

**Recommended**: Approach 2 (Unified Event Handler) for agent population, with a new Activity slice for tool call streaming.

#### Viewport Deep Dive

**Current Implementation** (Viewport.tsx:48-94):
```typescript
// Current: Item-based pagination
const visibleItems = items.slice(scrollOffset, scrollOffset + height);
```

**Problem**: `height` is number of items (messages), not lines. A message with 50 lines of code occupies same "height" as a one-line message. Result: scrolling jumps erratically.

**Fix Options**:

1. **Pre-calculate line heights**: Measure each message's rendered height, use for scroll calculations
   - Pro: Accurate scrolling
   - Con: Expensive, requires render-before-measure

2. **Virtual scrolling with line estimation**: Estimate lines per message, adjust on render
   - Pro: Performant
   - Con: Scroll position drift possible

3. **Ink's built-in scrolling**: Use `<ScrollArea>` if available in Ink 5
   - Pro: Native solution
   - Con: May not exist or have limitations

**Recommendation**: Option 2 with scroll position correction on content change.

#### Activity Slice Design

New store slice for real-time agent activity:

```typescript
// store/slices/activity.ts
interface ActivitySlice {
  // Map of agentId -> activity items
  activities: Record<string, ActivityItem[]>;

  // Add activity for an agent
  addActivity: (agentId: string, item: ActivityItem) => void;

  // Clear old activities (memory management)
  pruneActivities: (maxPerAgent: number) => void;
}

interface ActivityItem {
  id: string;
  timestamp: number;
  type: 'tool_call' | 'tool_result' | 'text_chunk' | 'status_change';
  data: {
    toolName?: string;
    toolInput?: Record<string, unknown>;
    text?: string;
    status?: AgentStatus;
  };
}
```

**Integration**: In `handleAssistantEvent`, after processing message content:
```typescript
// Extract tool_use blocks and add as activity
const toolUses = event.message.content.filter(b => b.type === 'tool_use');

    type: 'tool_call',
    data: { toolName: tool.name, toolInput: tool.input },
  });
});
```
